# Decision: hand-author the four generic OperatorMethodInputMap/OutputMap families, with a mandatory schema-consistency check

Date: 2026-07-06
Scope: Wave 5 SDK enabler — W5-S2 (contracts typed-I/O completion, the SDK half of the 1.0.0 typed-surface ruling)
Status: accepted

## Context

`OperatorTypedMethodId = OperatorMethodId` (`packages/contracts/src/generated/foundation-client-types.ts:597`)
already covers every method id, including `fleet.snapshot`, `fleet.list`,
`checkpoints.list/create/diff/restore`, and `sessions.search`/`sessions.detach`.
But `OperatorMethodInputMap`/`OperatorMethodOutputMap` had zero entries for
those seven methods, so `OperatorMethodInput<M>`/`OperatorMethodOutput<M>`
fell back to `{ [k: string]: unknown }` / `unknown` for all of them
(`foundation-client-types.ts:598-601`) even though `approvals.*` — a
precedent family reachable the same way — was fully typed.

`foundation-client-types.ts` carries the header comment "Generated by
scripts/export-foundation-artifacts.ts. Do not edit manually." That script
does not exist anywhere in this repository:

- no file at any path matching `export-foundation-artifacts*` in the working
  tree (only `scripts/refresh-contract-artifacts.ts` exists, and it never
  touches `foundation-client-types.ts` — it only regenerates
  `operator-contract.{ts,json}`, `operator-method-ids.ts`,
  `peer-contract.ts`, `peer-endpoint-ids.ts`, and `foundation-metadata.ts`);
- `git log --all -- '**/export-foundation-artifacts*'` returns nothing — the
  script has no history in this repository, meaning it was never committed
  here, not merely deleted;
- no `package.json` script or `bun run` reference names it anywhere in the
  monorepo.

It could not be located or restored. Per the brief's ruling
(restore-or-hand-author-with-consistency-check), the fallback path applies.

## Decision

Hand-author the seven missing `OperatorMethodInputMap`/`OperatorMethodOutputMap`
entries (`fleet.snapshot`, `fleet.list`, `checkpoints.list`,
`checkpoints.create`, `checkpoints.diff`, `checkpoints.restore`,
`sessions.search`, plus `sessions.detach` — eight methods, sixteen entries
counting input+output) directly in
`packages/contracts/src/generated/foundation-client-types.ts`, following the
exact TS-type-string conventions already used throughout that file (verified
against dozens of existing entries: property declaration order preserved,
enum values sorted alphabetically, optional-vs-required driven by the
schema's `required` array, `additionalProperties: true` rendered as
`({ ... } & { readonly [key: string]: unknown })`, nullable via
`anyOf: [X, {type:'null'}]` rendered `null | X`, and the fixed
`METADATA_SCHEMA` record-type literal repeated verbatim).

Rather than trust the hand transcription alone, a real (if narrowly scoped)
generator/checker was written: `scripts/check-foundation-io-types.ts`. It:

1. imports the actual method-catalog schema objects (source of truth) for
   the eight target methods from
   `packages/sdk/src/platform/control-plane/operator-contract-schemas-fleet.ts`
   and `operator-contract-schemas-runtime.ts` (`SHARED_SESSION_RECORD_SCHEMA`
   for `sessions.detach`'s output, since its descriptor uses
   `actionResultOutputSchema('session', SHARED_SESSION_RECORD_SCHEMA)` —
   identical to the already-typed `sessions.close`/`sessions.reopen`);
2. renders each schema to the same TS-type-string convention (a scoped
   schema-to-TS-string compiler — string/number/boolean/null, sorted
   enums, arrays with the observed parenthesization rules, nullable,
   plain objects with required/optional fields, `additionalProperties`,
   and the fixed metadata-record literal);
3. diffs the render against what is actually committed in
   `foundation-client-types.ts` for each of the sixteen map entries,
   the same "generate then diff the committed artifact" shape as
   `refresh-contract-artifacts.ts --check`;
4. throws (rather than silently guessing) on any schema construct it does
   not recognize.

`scripts/check-contract-artifacts.ts` (the `contracts:check` gate) now runs
this checker immediately after its existing `refresh-contract-artifacts.ts
--check` call, so `bun run contracts:check` is the one command that catches
drift in both generated surfaces. A standalone
`contracts:foundation-io:check` script entry was also added for direct
invocation.

As an independent cross-check (not just self-consistency of the same
renderer), `bunx api-extractor run --local` was run over the changed source;
its own TypeScript-driven `.d.ts` rollup expansion of the eight methods in
`etc/goodvibes-sdk.api.md` matches the hand-authored/rendered entries
field-for-field — a second, independently-computed confirmation that the
shapes are correct.

## Consequences

- `fleet.*`, `checkpoints.*`, `sessions.search`, and `sessions.detach` now
  resolve real typed I/O through `OperatorMethodInput`/`OperatorMethodOutput`
  instead of the generic fallback. `approvals.*` typing is unchanged
  (regression-checked: `contracts:check` passed with the full existing test
  suite green — 3376 tests, 0 failures).
- The consistency check is scoped to these eight methods only, not a general
  replacement for `export-foundation-artifacts.ts`. If a future method
  needs `foundation-client-types.ts` entries added by hand again, either
  extend `check-foundation-io-types.ts`'s `ENTRIES` list (cheap — one schema
  import + one array row) or restore/rebuild the real generator; this
  decision does not claim the generator problem is solved project-wide, only
  that this file's four target families cannot silently drift from their
  schemas going forward.
- Per the ship-vehicle note in the Wave-5 brief meta
  (`scripts/sdk-dev.ts:42-80` never overlays `node_modules/@pellux/goodvibes-contracts`
  — only the SDK dist), these generated shapes reach the webui's dev overlay
  only at the real `0.38.0 -> 1.0.0` npm pin-bump, not before. W5-TC's local
  bridge-type seam is the intended decoupling; this work does not change
  that.
- **`sessions.delete` was NOT added.** This worktree branched from SDK main
  @ `01ee9b3a`, before W5-S1 (parallel worktree). At this commit,
  `sessions.delete` is not a registered method id anywhere in the catalog
  (`operator-method-ids.ts` has no such entry, and
  `method-catalog-control-core.ts`'s spine-session block is still
  close/reopen/detach only) — there is no schema to type it against yet.
  Inventing a shape for a method that doesn't exist would itself be
  dishonest generation. When S1 lands `sessions.delete` (per its decision
  record's PREFERRED ruling), extending this checker to cover it is a
  ~5-line addition: import its input/output schema and add one row to
  `check-foundation-io-types.ts`'s `ENTRIES` array, then insert the rendered
  entries into `foundation-client-types.ts` the same way this change did for
  `sessions.detach`. Flagged for whoever serializes the S1+S2 land on the
  shared artifacts.

## Rejected alternatives

- **Leaving the four families generic.** Rejected — the 1.0.0 typed-surface
  ruling requires all four consumers to validate a fully-typed operator
  surface; the generic fallback is exactly the gap this brief exists to
  close.
- **Hand-authoring without a consistency check.** Rejected — hand types
  drift from schemas silently and exactly reproduce the "trust me" problem
  the missing generator already caused once.
- **Blocking on restoring/rewriting the full `export-foundation-artifacts.ts`
  generator.** Rejected as in-scope for this change — its surface (every
  method in the catalog, plus events, plus peer endpoints) is much larger
  than the eight methods this brief needs, and rebuilding it correctly is a
  separate, larger effort than the ruling requires right now. The scoped
  checker is deliberately named and structured so it does not pretend to be
  that generator.
