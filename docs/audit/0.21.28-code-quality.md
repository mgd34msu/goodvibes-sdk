## Findings

### [QA-01] Concentrated `as unknown as` casts in daemon HTTP client — HIGH
**File**: `packages/sdk/src/_internal/platform/runtime/transports/daemon-http-client.ts` (23 of 100 total — 23%)
**Issue**: 23 `as unknown as Record<string, unknown>` / `as unknown as readonly RuntimeTask[]` casts clustered in lines 124–358. They reinterpret operator-API responses into SDK domain shapes without runtime validation. If `operatorApi.sessions.list(...)` ever changes its element shape the casts will silently hand out malformed records to the gateway and normalize calls will paper over the drift.
**Recommendation**: Replace casts with explicit zod (or hand-rolled) guards at the boundary — one normalizer per response type — and return `unknown` from the operator client, making the guard mandatory. The `normalizeSharedSessionRecord(x as unknown as Record<string, unknown>)` pattern is the anti-pattern: the normalizer should accept `unknown` and the cast disappears.

### [QA-02] Non-null assertions in knowledge/store upsert paths — HIGH
**File**: `packages/sdk/src/_internal/platform/knowledge/store.ts:259-471` (25 occurrences)
**Issue**: Every `input.X!.trim()` site relies on a preceding `stableText(input.X)` guard, but TS control-flow narrowing does not understand `stableText`. The `!` assertion bypasses the type system. If `stableText` is ever weakened or a caller skips it, the runtime throws on `undefined.trim()`.
**Recommendation**: Refactor `stableText` to return the trimmed string-or-null (type-guard with return value) and use `const t = stableText(input.title); ...(t ? { title: t } : {})`. Eliminates both the `!` and the second `trim()` call.

### [QA-03] Repeated `as unknown as Record<string, unknown>` normalizer pattern — MEDIUM
**Files**: `edit/match.ts` (6), `daemon/http/router-route-contexts.ts` (5), `providers/anthropic.ts` (4), `media/builtin-image-understanding.ts` (4), `daemon/facade.ts` (3), plus ~15 files at 2 each
**Issue**: The same "normalize wire-shape to internal shape" idiom is copy-pasted across ~20 files. Each site is 1–6 casts, but the aggregate is ~80% of the project's unsafe-cast debt.
**Recommendation**: Extract a shared `asRecord(x: unknown): Record<string, unknown> | undefined` helper in `packages/sdk/src/_internal/platform/utils/` and make it a runtime check (`typeof x === 'object' && x !== null && !Array.isArray(x) ? x as Record<…> : undefined`). A single validated helper replaces ~60 cast sites and gains a null fallback the casts currently lack.

### [QA-04] God-object accretion in `providers/registry.ts` — MEDIUM
**File**: `packages/sdk/src/_internal/platform/providers/registry.ts` (823 LOC, ~79 method-shaped definitions)
**Issue**: Highest method density in the codebase. Registry carries provider catalogue, health checks, model-limit resolution, credential resolution, and runtime mutation. Testable but grows with every new provider.
**Recommendation**: Split along the four clear axes (catalogue, credentials, limits, health). The module already has `model-limits.ts` and `builtin-catalog.ts` siblings; the registry class can delegate rather than absorb.

### [QA-05] God-object in automation `manager-runtime.ts` — MEDIUM
**File**: `packages/sdk/src/_internal/platform/automation/manager-runtime.ts` (740 LOC, ~68 methods)
**Issue**: Second-densest class. Also holds the single snake_case leak (`slots_total`, `slots_in_use`, etc. at line 638) — these are wire-format fields intentionally not camelCased but would read better as `{ slotsTotal: n } satisfies SchedulerCapacityWire` with a single serialization point.
**Recommendation**: Extract scheduler-capacity reporting into a focused helper; clarifies the naming contract and reduces the class footprint.

### [QA-06] `session-broker.ts` — large, but reasonably cohesive — LOW
**File**: `packages/sdk/src/_internal/platform/control-plane/session-broker.ts` (1052 LOC, 56 methods)
**Issue**: Largest non-generated TS file. All methods relate to one concern (shared-session lifecycle) and the file has a dedicated test suite, so god-object risk is mitigated.
**Recommendation**: No split required, but watch the trajectory — GC, persistence, and event-publishing concerns are starting to mingle.

### [QA-07] `scan-modes.ts` has no targeted test — MEDIUM
**File**: `packages/sdk/src/_internal/platform/tools/analyze/scan-modes.ts` (762 LOC, 10 exported `run*` functions)
**Issue**: Ten distinct scan-mode entry points (impact, dependencies, deadCode, security, coverage, bundle, surface, preview, permissions, envAudit), each >50 LOC, with no `scan-modes.test.ts` present. Grep of `test/` matched zero files.
**Recommendation**: Add fixture-based tests for at least the security and deadCode modes — these are the highest-risk outputs. The other modes are lower-risk read-only reporters.

### [QA-08] `wrfc-controller.ts` lacks targeted test — MEDIUM
**File**: `packages/sdk/src/_internal/platform/agents/wrfc-controller.ts` (748 LOC, 51 methods)
**Issue**: Core WRFC orchestration with no file named `wrfc-controller.test.ts`. The wrfc-gates sibling has empty `catch {}` on `proc.kill()` (lines 47, 55) which is fine but adjacent concerns are untested.
**Recommendation**: Add a controller-level test exercising the happy path plus one gate-failure path. Currently covered only transitively.

### [QA-09] `control-plane/gateway.ts` lacks targeted test — MEDIUM
**File**: `packages/sdk/src/_internal/platform/control-plane/gateway.ts` (757 LOC)
**Issue**: No `gateway.test.ts` in `test/`. Cache-invariants and session-broker are well tested but the gateway aggregating them is not.
**Recommendation**: Add a smoke test covering gateway construction + one end-to-end emit + one invariant assertion.

### [QA-10] Deep nesting in `tools/write/index.ts` — LOW
**File**: `packages/sdk/src/_internal/platform/tools/write/index.ts`
**Issue**: 46 lines at ≥16 leading whitespace chars (>4 levels of nesting). Highest in the codebase by a wide margin (next is <10).
**Recommendation**: Flatten with early returns or extract the inner block into a helper. Nesting depth correlates strongly with defect rate.

### [QA-11] Hardcoded localhost/port URLs — LOW
**Files**: `hooks/workbench.ts:161` (`http://localhost:3000/hooks`), `media/builtin-generation-providers.ts:478` (`http://127.0.0.1:8188`), `providers/builtin-catalog.ts:297` (`http://localhost:4000/v1`), `providers/builtin-catalog.ts:310` (`http://localhost:3000/v1`), 70 further localhost occurrences.
**Issue**: Local-dev defaults embedded in library code. Mostly acceptable (LM Studio, Ollama, Comfy conventions) but should be named constants for discoverability.
**Recommendation**: Hoist the well-known-local-service URLs to `_internal/platform/providers/well-known-endpoints.ts` so they are a searchable, testable catalog.

### [QA-12] Mixed `.then()` and `await` in 10+ files — LOW
**Files**: `discovery/scanner.ts`, `daemon/facade-composition.ts`, `channels/builtin-runtime.ts`, `tools/shared/process-manager.ts`, `hooks/dispatcher.ts`, `providers/custom-loader.ts`, `providers/registry.ts`, `automation/manager-runtime.ts`, `core/orchestrator-context-runtime.ts`, `automation/manager-runtime-delivery.ts`
**Issue**: Same file mixes `await fn()` with `otherFn().then(handler)` — usually the `.then()` is a fire-and-forget side-effect the author did not want to `await`. Readers must now track which is which.
**Recommendation**: Prefer `void (async () => { ... })()` or an explicitly named `queueMicrotask` for fire-and-forget; keep `.then()` out of files that otherwise use `await`. Grep `.then(` currently returns 52 hits project-wide.

### [QA-13] Commented-out migration bodies — LOW
**Files**: `runtime/contracts/migrations/{task-record,session,event-envelope,runtime-state,plugin-manifest}.ts` lines 29–34 in each
**Issue**: Each migration file carries an identical commented `// const d = data as Record<string, unknown>; return { ...d, X: d['X'] ?? default };` block showing what a v1→v2 migration would look like. Five near-identical stubs.
**Recommendation**: Replace with one JSDoc block in `migrations/README.md` (or `migrations/index.ts` as a `@see` reference). The duplication adds visual noise without educational value once you've read the first one.

### [QA-14] Error-shape convention divergence — LOW
**Counts**: `code: '...'` string-literal pattern: 6 occurrences. `hint: '...'` pattern: 66 occurrences. `throw new Error(...)`: 462. `throw new <Named>Error(...)`: 571.
**Issue**: The audit brief cites `{code, message, hint}` as the canonical error shape. The `hint` field is used 66 times but `code` (as literal) only 6 times. Named `*Error` subclasses (571) carry the code implicitly via class identity, so the convention is really "subclass-driven code + optional hint on throw site" — not the tuple described in the brief.
**Recommendation**: Either (a) document the actual convention in `packages/errors/README.md` as the source of truth, or (b) align by adding `readonly code` to each named error subclass so the shape is uniform across both sites.

### [QA-15] Empty catches around best-effort cleanup — INFO
**Files**: `tools/shared/overflow.ts:151,167,180,182`; `control-plane/session-broker.ts:121`; `agents/wrfc-gates.ts:47,55`
**Issue**: All 7 empty catches wrap `statSync`, `unlinkSync`, `readFileSync`, unsubscribe callbacks, or `proc.kill()`. These are deliberate best-effort swallows and each is on a single line — the intent is legible.
**Recommendation**: No change needed. If consistency matters, add a single `ignoreError(() => proc.kill())` helper and call it from the 7 sites; turns intent into a name.

### [QA-16] JSDoc coverage on public APIs — INFO
**Observation**: `client.ts`, `auth.ts`, `observer/index.ts` have thorough JSDoc (observer file has `@example` blocks). The 9 transport/subpath entry files (`transport-http.ts`, `transport-realtime.ts`, `peer.ts`, `operator.ts`, `daemon.ts`, etc.) are re-export shims so JSDoc is not expected there. Public surface documentation appears solid.
**Recommendation**: No action.

### [QA-17] `client.ts` single non-null assertion — INFO
**File**: `packages/sdk/src/client.ts:328` — `() => options.tokenStore!.getToken()`
**Issue**: Inside an arrow created only after checking `options.tokenStore` exists, but across a closure boundary TS narrowing is lost so `!` is used.
**Recommendation**: Capture the narrowed value (`const store = options.tokenStore; if (!store) return ...; return () => store.getToken();`). Trivial fix and removes the last non-null assertion from the public entry point.

## Closing assessment

The codebase scores well above average on the primary type-safety axes: zero `any`, zero TODOs, vanishing `@ts-ignore` footprint, strong readonly discipline (8.8:1 over `let`). The work to be done is structural, not defensive:

1. **QA-01 + QA-03** together: introduce one `asRecord` helper and make `daemon-http-client.ts` call validated normalizers on `unknown`. Eliminates ~60 of the 100 unsafe casts.
2. **QA-02**: refactor `stableText` to return the trimmed value. Removes 25 of the 49 non-null assertions.
3. **QA-07 / QA-08 / QA-09**: add dedicated test files for `scan-modes`, `wrfc-controller`, `gateway`. The integration layer is strong but file-level regression safety is uneven.

Everything else is polish.
