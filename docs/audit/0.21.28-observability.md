**Bus architecture** (`packages/sdk/src/_internal/platform/runtime/events/index.ts`):
`RuntimeEventBus.emit()` dispatches synchronously to a snapshotted `Set` of
per-type and per-domain listeners. Catches handler errors, logs via `logger.error`.
No queuing, no back-pressure, no microtask hop.

**Metrics**: `RuntimeMeter` (`runtime/telemetry/meter.ts`) is a pure
in-memory counter/histogram/gauge registry. It is instantiated but the grep for
`counter.add|histogram.record|gauge.set` across the runtime returns **zero
production call-sites** â€” the meter is exposed but unused.

**Telemetry API**: `TelemetryApiService` (`runtime/telemetry/api.ts`) subscribes
to all 27 domains, bounds event/error/span arrays, derives synthetic spans for
turn/task/agent/tool lifecycles. Good architecture, but the API itself is the
*only* source of aggregate/metrics snapshots â€” no independent metric emissions
from code paths.

**OTLP exporters**: `runtime/telemetry/exporters/otlp.ts` (+`queue.ts`) own the
push path. Inbound OTLP ingest was explicitly rejected per `decisions.json`
(`otlp-no-post-ingest`, 2026-04-19) â€” GET consumer-pull only.

## Findings

### [OBS-01] HTTP request access log is entirely silent â€” no events, no logger â€” HIGH
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:333-370` (`handleRequest`), `packages/sdk/src/_internal/platform/daemon/http/router.ts:185+` (`handleRequest`)
**Gap**: `handleRequest` resolves URL, performs rate-limit check, runs auth, dispatches, and returns a `Response`. No `logger.info` per request, no emission into the `routes` or `control-plane` domain, no latency measurement, no status-code capture. The only logging in this file is server start/stop.
**Impact**: Cannot answer "which requests are hitting the daemon", "what's the p95 latency", "what's the 4xx/5xx rate", "which IP is being rate-limited", without attaching a live SSE subscriber. No counter/histogram populated â€” a full access log blind spot.
**Recommendation**: Emit a `ControlPlaneEvent` (e.g. `REQUEST_RECEIVED`, `REQUEST_COMPLETED`) around `handleRequest`, carrying `{ method, path, status, durationMs, clientIp, principalKind }`. Populate a `requests_total` counter labelled by `method,status_class` and a `request_duration_ms` histogram labelled by `route_group`. Log at `info` with the same fields.

### [OBS-02] Login/auth success and failure are not emitted anywhere â€” HIGH
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:372-400` (`handleLogin`), `packages/sdk/src/_internal/platform/security/user-auth.ts` (authenticate, createSession)
**Gap**: `handleLogin` returns `401` for bad credentials and `200` with a cookie for success â€” no `logger.info/warn`, no `security` or `control-plane` domain emission. `user-auth.ts`'s `authenticate` path returns null silently on bad passwords. The `SecurityEvent` domain exists but no code emits auth-specific events from this path.
**Impact**: Cannot detect brute-force, cannot audit admin login timelines, cannot answer "when did user X last authenticate" without scraping the cookie issuance side effect.
**Recommendation**: Emit `AUTH_SUCCEEDED { username, sessionId, clientIp, method: 'password'|'cookie'|'token' }` and `AUTH_FAILED { usernameAttempted, clientIp, reason }` into the `security` domain. Log at `info` for success, `warn` for failure. Populate `auth_failures_total` counter labelled by reason.

### [OBS-03] Outbound HTTP calls (providers, webhooks, voice, LSP downloader) emit nothing â€” HIGH
**File(s)**: 50+ files with direct `fetch(` calls, including `providers/anthropic.ts`, `providers/gemini.ts`, `providers/openai-codex.ts`, `providers/github-copilot.ts`, `integrations/slack.ts`, `integrations/discord.ts`, `integrations/github.ts`, `integrations/ntfy.ts`, `voice/providers/*.ts`, `media/builtin-generation-providers.ts`, `state/memory-embedding-http.ts`, `web-search/providers/*.ts`, `tools/fetch/runtime.ts`
**Gap**: Every outbound HTTP call-site is a bare `await fetch(url, {...})`. No span, no counter, no per-provider latency histogram, no per-endpoint error tally. Errors are typically caught and converted to domain errors but the wire-level observation is absent.
**Impact**: When an Anthropic or Gemini call is slow/errors, nothing tells you whether it was the network, the provider, or our code. No provider-latency dashboards are possible from the emitted signal set.
**Recommendation**: Introduce a shared `instrumentedFetch(req, { domain, endpoint, provider })` helper in `utils/fetch-with-timeout.ts` that records `http_client_duration_ms{provider,endpoint,status}` histogram, `http_client_errors_total{provider,endpoint,cause}` counter, and emits `PROVIDER_REQUEST_STARTED`/`PROVIDER_REQUEST_COMPLETED` for LLM calls.

### [OBS-04] LLM provider calls emit completion but not start/latency/cost â€” HIGH
**File(s)**: `packages/sdk/src/_internal/platform/runtime/emitters/turn.ts:70-86` (`emitLlmResponseReceived`), provider files `packages/sdk/src/_internal/platform/providers/{anthropic,gemini,openai-codex,github-copilot,anthropic-compat,llama-cpp,ollama}.ts`
**Gap**: Only `LLM_RESPONSE_RECEIVED` is emitted, with tokens and content but no `durationMs`, no `startedAt`, no `retryCount`, no `usdCostEstimate`, no `promptCacheHit`. There is no paired `LLM_REQUEST_STARTED`. The providers domain has a broader vocabulary (`ProviderEvent`) but the emitter call-sites are absent from provider files (grep `complete(|generate(|invokeProvider|provider.send` at `providers/` returns zero matches â€” call-sites are named differently or fully inlined).
**Impact**: Cannot compute provider latency percentiles, cannot detect retry storms, cannot bill/audit usage per session without reconstructing from partial signals.
**Recommendation**: Emit `LLM_REQUEST_STARTED { turnId, provider, model, messageCount, estimatedInputTokens }` before dispatch and extend `emitLlmResponseReceived` with `{ durationMs, retryCount, providerRequestId, finishReason, cacheReadTokens, cacheWriteTokens, estimatedCostUsd }`. Populate `llm_latency_ms{provider,model}` histogram and `llm_tokens_total{provider,model,kind}` counter.

### [OBS-05] Tool-call result payloads carry optional `result: unknown` â€” privacy/cardinality risk â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/emitters/tools.ts:75` (`emitToolSucceeded`), `:84` (`emitToolFailed`)
**Gap**: `data.result?: unknown` â€” tool outputs that may contain file contents, exec stdout (potentially with secrets/env), or full LLM-generated code are emitted raw. `TelemetryApiService.sanitizeRecord` exists but default view is 'safe' and sanitization rules are not visible in the emitter boundary.
**Impact**: Raw secrets can transit to OTLP exporters, SSE subscribers, and local-ledger without opt-in. Privacy incident risk.
**Recommendation**: Narrow the `result` type to a `ToolResultSummary { kind, byteSize, preview?: string }` at the emission boundary. Require explicit `includeRawResult: true` flag per session config. Audit `sanitizeRecord` at `api-helpers.ts` for coverage.

### [OBS-06] Turn prompt is emitted raw in `TURN_SUBMITTED` â€” privacy â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/emitters/turn.ts:15`, `emitters/session.ts` (no prompt), `turn.ts` STREAM_DELTA content field `:55`, TURN_COMPLETED response `:121`
**Gap**: `TURN_SUBMITTED.prompt: string` and `TURN_COMPLETED.response: string` and `STREAM_DELTA.{content, accumulated, reasoning}` all carry full user/assistant text. No explicit redaction, no length cap at the emitter layer.
**Impact**: Any telemetry subscriber sees the full prompt/response stream, including secrets the user pasted, regardless of the 'safe'/'raw' view. OTLP export to third-party collectors leaks PII.
**Recommendation**: Apply length cap (e.g. 256 chars + hash) at emitter boundary for non-'raw' view. Document the 'raw' view as operator-only. Consider a runtime-config `telemetry.redactPromptContent` flag (default true).

### [OBS-07] OTLP exporter uses raw `console.error/warn` instead of structured logger â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/telemetry/exporters/otlp.ts:161,177,194,239,243`; `exporters/queue.ts:133,228,237`
**Gap**: Export failures, drops, and overflow warnings bypass `logger` and go straight to `console.(warn|error)`. Messages contain numeric counts but no traceId/spanId, no structured shape for downstream log-aggregators.
**Impact**: These events are invisible in the activity.md log, do not show up in telemetry viewers, and cannot be re-emitted as `ops` domain events for operator dashboards. Ironically, the OTLP exporter's own failures are the least observable in the system.
**Recommendation**: Replace each `console.*` with `logger.(warn|error)(message, { spanCount, attempts, endpoint, errorCategory })` and emit an `OpsEvent` (e.g. `OTLP_EXPORT_FAILED`) into the `ops` domain so subscribers can react.

### [OBS-08] WorkspaceSwap emits 3 of 4 needed events â€” missing SWAP_FAILED â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/workspace/workspace-swap-manager.ts:108-118` (mkdir failure path), `:121-127` (rerootStores failure path)
**Gap**: The manager emits `WORKSPACE_SWAP_STARTED`, `WORKSPACE_SWAP_REFUSED`, `WORKSPACE_SWAP_COMPLETED`. When mkdir or `rerootStores` throws, it returns `{ ok: false, code: 'INVALID_PATH' }` **without emitting a failure event** â€” so a subscriber that saw `_STARTED` will never see terminal resolution.
**Impact**: Any downstream dashboard or SSE subscriber watching for swap lifecycle sees a swap go into flight and never resolve â€” looks like a hang.
**Recommendation**: Add a `WORKSPACE_SWAP_FAILED` event to `WorkspaceEvent` union and emit it on both catch blocks before returning the error. Include `{ from, to, code, reason }`.

### [OBS-09] `persistedInDaemonSettings = false` failure is silently swallowed â€” LOW
**File(s)**: `packages/sdk/src/_internal/platform/workspace/workspace-swap-manager.ts:128-136`
**Gap**: `try { writeDaemonSetting... } catch { /* Non-fatal â€” swap succeeded but persistence failed */ }`. The boolean ends up on the `WORKSPACE_SWAP_COMPLETED` payload, but no logger call, no separate event, and the error object is lost.
**Impact**: Silent loss of the underlying error. Cannot diagnose why daemon settings write failed (permissions, disk full, path). On next daemon restart, workingDir will not persist.
**Recommendation**: Log `logger.warn('[WorkspaceSwap] daemon settings persistence failed', { error: summarizeError(err) })` in the catch.

### [OBS-10] SessionBroker uses `console.warn/error` for runtime bus wiring failures â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/control-plane/session-broker.ts:145` (`attachRuntimeBus called more than once`), `:163,181` (completeAgent error on AGENT_COMPLETED/AGENT_FAILED/AGENT_CANCELLED)
**Gap**: These are the control plane's core reconciliation errors â€” agent termination failing to propagate to session state. They are printed to console with no structure, no correlation IDs.
**Impact**: On production, these errors do not hit the activity log, do not emit a `session` or `agents` domain event, and are lost on any `ChildProcess`-wrapped daemon where stderr is not captured.
**Recommendation**: Replace with `logger.error('[SharedSessionBroker] completeAgent on AGENT_COMPLETED', { agentId, sessionId, error: summarizeError(err) })` and emit a `SessionEvent` like `SESSION_RECONCILE_FAILED` for subscriber visibility.

### [OBS-11] Silent catch blocks on 28 files mask failure paths â€” MEDIUM
**File(s)**: 48 matches across 28 files. Notable: `control-plane/gateway-web-ui.ts:589,612,628`, `tools/exec/runtime.ts:225,324,383`, `tools/shared/overflow.ts:151,167,180`, `tools/shared/process-manager.ts:138,142,196`, `daemon/surface-delivery.ts:455,480,484`, `hooks/runners/command.ts:47,56`, `plugins/loader.ts:288,309`, `config/manager.ts:258,280`, `adapters/slack/index.ts:144,166`, `adapters/discord/index.ts:132,145`, `agents/completion-report.ts:82,110`, `state/file-watcher.ts:211`
**Gap**: Empty or comment-only `catch` blocks (e.g. `catch {}`) with the error object discarded. While some are intentional ("best-effort" paths), the pattern is indiscriminate â€” there is no `logger.debug` breadcrumb of what was swallowed.
**Impact**: Debugging intermittent tool/hook/plugin/adapter failures requires adding logs and reproducing. No forensic trail.
**Recommendation**: Default to `logger.debug('[<module>] swallowed non-fatal error', { error: summarizeError(err) })` in each. Carve out explicit no-log cases with a comment explaining why. Consider a lint rule against `catch {}` / `catch () {}`.

### [OBS-12] RuntimeMeter is defined and never called in production â€” HIGH
**File(s)**: `packages/sdk/src/_internal/platform/runtime/telemetry/meter.ts` (174 lines), `telemetry/index.ts:23,51`
**Gap**: Grep for `\b(counter|histogram|gauge)\.(add|record|set)\b` across `packages/sdk/src/_internal` returns zero call-sites outside meter.ts's self-tests. The public `RuntimeMeter.counter()/histogram()/gauge()` APIs are defined but unused by the runtime.
**Impact**: No metrics are actually accumulated. `TelemetryApiService.getAggregates()` computes counts from the event array at read time, which is fine for low-volume observability but cannot answer request-rate, latency-percentile, or provider-cost questions.
**Recommendation**: Populate from the domain bridge or at emitter boundaries. Minimum seed: `events_total{domain,type}`, `llm_latency_ms{provider,model}`, `tool_duration_ms{tool,phase}`, `request_duration_ms{route_group,status_class}`, `queue_size{queue}` (OTLP export queue + scheduler).

### [OBS-13] Event bus listener errors use `logger.error` but do not degrade the system â€” LOW
**File(s)**: `packages/sdk/src/_internal/platform/runtime/events/index.ts:170-187`
**Gap**: Listener errors are caught and logged, but there is no counter, no circuit-breaker for a listener that errors every time, and no subscriber-ID-based diagnosis (each listener is anonymous).
**Impact**: A misbehaving subscriber (e.g. TelemetryApiService, an extension) can flood activity.md with errors on every emit with no signal to the operator beyond log volume.
**Recommendation**: Track `listener_errors_total{eventType}` counter; after N consecutive errors, `logger.error` once with a deduplication key; emit an `OpsEvent.LISTENER_MISBEHAVING`.

### [OBS-14] Event bus dispatch is fully synchronous â€” blocking emitters â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/events/index.ts:155-189`
**Gap**: `emit()` iterates all handlers inline. A slow subscriber (e.g. a synchronous OTLP serializer or a noisy logger.error) blocks the next listener and the emitter's call stack. No microtask hop (`queueMicrotask`) or `setImmediate` isolation.
**Impact**: One slow subscriber can starve the event loop and back-propagate latency into the emitting code path (turns, tool execution). No unbounded queue risk exists, but head-of-line blocking does.
**Recommendation**: Dispatch per-listener via `queueMicrotask` or a single microtask per emit. Alternatively introduce a `priority` tier (e.g. internal store vs external subscribers) where external listeners go through a bounded queue with drop-oldest semantics.

### [OBS-15] No correlation ID in logger payloads â€” logs cannot be stitched to events â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/utils/logger.ts` (ActivityLogger)
**Gap**: `ActivityLogger.write(level, message, data)` writes `[ISO] [LEVEL] message` + JSON `data`. There is no mandatory `traceId` / `sessionId` / `turnId` field â€” it's ad-hoc per call-site. Many call-sites omit the correlation completely (e.g. `logger.info('HttpListener started', { port, host })`).
**Impact**: Cross-referencing an error in activity.md against the event stream requires grep by timestamp. No traceId propagation means a tool failure log line cannot be linked to the turn/agent/session that caused it.
**Recommendation**: Extend `logger.write` to accept a required `correlation?: { traceId?, sessionId?, turnId?, agentId? }` argument (type-enforced). Add an `AsyncLocalStorage`-backed ambient correlation context for common paths.

### [OBS-16] Error cause chains are flattened to strings â€” no root-cause â€” MEDIUM
**File(s)**: `summarizeError` is used everywhere, e.g. `events/index.ts:172,185`; `daemon/http-listener.ts:334`; `config/manager.ts`; many emitter payloads declare `error: string`.
**Gap**: All error payloads in `TurnEvent.TURN_ERROR`, `ToolEvent.TOOL_FAILED`, `TransportEvent.TRANSPORT_TERMINAL_FAILURE`, `WorkspaceEvent.WORKSPACE_SWAP_REFUSED` use `error: string` (see emitters/tools.ts:84, turn.ts:133, transport.ts:75). The underlying `.cause` chain (ES2022 `Error.cause`) is lost at emission boundary.
**Impact**: "Provider call failed: connect ETIMEDOUT" hides whether this was DNS, TLS, or HTTP layer. `normalizeError` in `api-helpers.ts` does richer work but the raw emitter types do not preserve it.
**Recommendation**: Replace `error: string` with `error: NormalizedError` (the type already exists in `utils/error-display.ts`) across all failure emitters so cause chains, categories, and provider attribution flow through.

### [OBS-17] No telemetry for OTLP ingest â€” but by design (decision: otlp-no-post-ingest) â€” INFO

### [OBS-18] Circuit-breaker / retry / backoff emit nothing â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/transport-http/retry.ts` (6 refs), `backoff.ts` (7 refs), `reconnect.ts` (2 refs), `sdk/src/_internal/platform/utils/retry.ts` (3 refs), `runtime/remote/transport-contract.ts` (6 refs)
**Gap**: The retry and backoff machinery computes delays, increments attempts, and eventually gives up â€” none of that emits into the `transport` domain or logs at `warn`. `TransportEvent.TRANSPORT_RECONNECTING` exists in the emitter vocabulary but no grep found a call-site that uses it from the retry path.
**Impact**: Users see connection hang-then-recover with no dashboard signal. Cannot answer "how often does the peer transport have to reconnect?" without wiring a custom probe.
**Recommendation**: Wire `utils/retry.ts` to emit `TRANSPORT_RECONNECTING { transportId, attempt, maxAttempts }` per retry and `TRANSPORT_DEGRADED` on first failure. Counter `retries_total{operation,cause}`.

### [OBS-19] SSE/WebSocket open/close/error not emitted from server side â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/telemetry/api.ts:~350` (`createStream` â€” telemetry SSE producer), many `packages/transport-http/src/{sse,sse-stream}.ts` client consumers, `daemon/http/router.ts` telemetry-routes dispatch

### [OBS-21] Peer/companion pairing events absent â€” MEDIUM

### [OBS-22] High-cardinality attribute risk: session/turn/agent IDs flowing into telemetry â€” LOW

**Gap**: `buildAttributes` folds sessionId, turnId, agentId, taskId into span attributes. For ops metrics these are OK as trace attributes but dangerous as *metric labels*. Since `RuntimeMeter` is unused today this is latent, but when OBS-12 is resolved, avoid labeling counters/histograms with session/turn/agent IDs.
**Impact**: Future metric-label explosion if OBS-12 is implemented naively.
**Recommendation**: Establish the metric-label allowlist up front (e.g. only `domain`, `type`, `provider`, `model`, `tool`, `status_class`, `route_group`) â€” reject IDs at the meter boundary.

### [OBS-23] Channel policy change events are typed but do not cover all mutation points â€” LOW

**Recommendation**: Audit each `policyManager.set(...)` / `bindings.upsert(...)` call-site and ensure paired emission.

### [OBS-24] No explicit token redaction in error payloads â€” MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/auth/oauth-client.ts`, `auth/auto-refresh.ts`, `auth/token-store.ts`, `security/user-auth.ts`
**Gap**: Error messages from `fetch` on auth-refresh paths can include bearer tokens if the server echoes the auth header (uncommon but possible). No unified redactor applied before `logger.error(..., { error: summarizeError(err) })`.
