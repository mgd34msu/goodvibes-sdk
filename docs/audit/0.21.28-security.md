# Security Red-Team Audit — goodvibes-sdk 0.21.28 (commit 395fb32)

**Date**: 2026-04-20
**Scope**: Adversarial review of the full SDK source tree (`packages/*/src/**`).
**Auditor**: automated code review (static inspection only — no live exploitation).
**Verification**: reads against canonical source under `packages/` (daemon-sdk, sdk internals, transports, contracts).

---

## Methodology

- Discovery via precision_grep/precision_read across all packages.
- Cross-check against `.goodvibes/memory/failures.json` (F3 operator-tokens, F7 OTLP ingest).
- Focused attack surfaces: HTTP listener, daemon HTTP routes (OTLP, webhook, companion chat, local-auth), process spawning, outbound fetch / SSRF, filesystem identity writes (tokens, auth-users), supply chain.
- No live traffic generated. All findings are from source reading.

## Findings

### [SEC-01] Auth-user store persisted without 0600 mode — password-hash disclosure — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/security/user-auth.ts:102-106`, same file `:385` (persist on add/delete/rotate)
**Attack**: `writeBootstrapUsers` calls `writeFileSync(filePath, …, 'utf-8')` with no `mode` option and no follow-up `chmodSync`. Default mode is `0644` (world-readable on typical POSIX umasks). The file contains scrypt password hashes + salts for the `admin` user (and any added users). Any unprivileged local user on the host can read `auth-users.json`, then run offline scrypt brute-force against the stored hash with no rate limit.
**Impact**: Offline password cracking of the daemon admin account. Compare with the adjacent `writeBootstrapCredentialFile` (`:122`) which correctly chmods `0600` — and the post-0.21.28 invariant that identity state under daemon-home must be `0600` (see failures.json `F3-operator-tokens-global-only` + `companion-token.ts:82-83` applying the same rule).
**Recommendation**: Apply `{ mode: 0o600 }` in `writeBootstrapUsers` and follow with `chmodSync(filePath, 0o600)` (write-then-chmod, matching `writeOperatorTokenFile` in `daemon-home.ts:162-170`). Consider write-to-tmp-then-rename for atomicity.

### [SEC-02] Daemon-home migration `safeCopy` does not enforce 0600 on copied identity files — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/workspace/daemon-home.ts:237-249` (`safeCopy`), invoked at `:132-140` for `auth-users.json` and `auth-bootstrap.txt`
**Attack**: On first 0.21.19+ startup, the migration copies `~/.goodvibes/tui/auth-users.json` and `~/.goodvibes/tui/auth-bootstrap.txt` into `~/.goodvibes/daemon/`. `copyFileSync` preserves the *source* mode. If the legacy TUI-era files were ever written world-readable (which they were, since `writeBootstrapUsers` still does — see SEC-01), the migration propagates that weakness into the new canonical location. No post-copy `chmod 0600` is applied.
**Impact**: A fresh 0.21.19+ migration keeps password hashes and the plaintext bootstrap password world-readable on the filesystem.
**Recommendation**: After each `safeCopy` for identity files, explicitly `chmodSync(dest, 0o600)`. Better: wrap into a dedicated `safeCopyIdentity` helper that enforces mode.

### [SEC-03] `/login` bypasses rate limiter and CORS-origin check — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:335-339` (dispatch) vs `:343-352` (rate limiter / origin check fall *after*)
**Attack**: `handleRequest` short-circuits `POST /login` *before* the CORS-origin check and *before* `rateLimiter.check(clientIp)`. Password authentication goes directly to `userAuth.authenticate` with scrypt verification on every request. An attacker who can reach the listener can issue unlimited `/login` attempts. Combined with SEC-01 this narrows search space further, but even standalone it permits online password guessing at CPU-bound speed (scrypt cost is the only throttle).
**Impact**: Online brute-force of the admin password with no per-IP cap and no origin filtering. When `httpListener.hostMode=network` (binds 0.0.0.0), brute force is reachable across the LAN.
**Recommendation**: Move the rate-limiter and origin check above the `/login` short-circuit. Additionally apply a stricter per-username failed-attempt lockout (e.g. exponential backoff after 5 failures, tracked in `UserAuthManager`).

### [SEC-04] Historical install-time minimatch fetch without integrity verification — MEDIUM
**Current status**: Superseded in 0.25.1 by the graph-level
`vendor/bash-language-server` patch. Current source keeps Bash LSP bundled and
does not mutate `minimatch` during install.
**Attack**: The removed pre-0.25.1 install-time mitigation fetched a minimatch
tarball and extracted it into the consumer dependency tree without a pinned
digest. If the registry response had been tampered with, downstream consumers
could have loaded attacker-controlled JavaScript.
**Impact**: Supply-chain compromise via malicious tarball substitution would
execute attacker JS in downstream consumers.
**Recommendation**: Use the current graph-level Bash LSP vendor patch. The
mitigation is represented in the published package manifest and audited as part
of the dependency graph.

### [SEC-05] `/webhook` and all daemon JSON-body routes have no request-size cap — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:321-327` (`parseJsonBody` — bare `req.json()` with no size limit), daemon HTTP router JSON bodies generally
**Attack**: `parseJsonBody` calls `await req.json()` without first checking `content-length` or streaming with a byte cap. OTLP ingest explicitly caps at 4 MiB (see `telemetry-routes.ts:120` — the resolved-in-0.21.x mitigation of F7), but the `/webhook` route, all `/api/*` POST/PATCH routes, and the companion-chat routes do not. Any authenticated (or, at `/login`, unauthenticated-but-rate-limited-eventually) client can submit an arbitrarily large JSON body and force the daemon to allocate memory during parse.
**Impact**: Memory exhaustion DoS. Attacker must pass auth for most routes, but `/webhook` is configured by `danger.httpListener` + bearer, and any operator token holder (SEC-03 bootstrap) can use the vector against the daemon process.
**Recommendation**: Extract `parseOtlpBody`'s size gate (arrayBuffer + byte check + `413`) into a shared `parseJsonBodyWithCap(req, maxBytes)` and apply it uniformly to every state-changing route. Default cap: 1 MiB for control-plane routes, 10 MiB for media/artifact ingest routes.

### [SEC-06] Companion-chat rate limiter maps are unbounded — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/companion/companion-chat-rate-limiter.ts:94-96` (map declarations), `:183-194` (`cleanup()` is opt-in, not periodic)
**Attack**: `CompanionChatRateLimiter` stores a `Bucket` entry per distinct `sessionId` and per distinct `clientId`. `cleanup()` is only called from `companion-chat-manager.ts:188` (on specific lifecycle events, not a periodic timer). A token-holding attacker can create N distinct sessions or submit N distinct `clientId` headers; each call allocates a new Map entry whose timestamps array sticks around until `cleanup()` runs. Unlike the HTTP `RateLimiter` in `http-listener.ts` (which has an LRU cap `RATE_MAX_ENTRIES = 10_000` and periodic sweep via `setInterval`), the companion limiter has neither.
**Impact**: Slow-growth memory DoS from an authenticated client. Amplifies the SEC-05 concern. The rate-limit itself still protects against message-flood per-session — but the limiter's own bookkeeping is a leak.
**Recommendation**: Add an LRU cap + background sweep mirroring `RateLimiter` in http-listener.ts. Or call `cleanup()` from the per-minute tick that already exists in the chat manager.

### [SEC-07] `HttpListener` CORS origin check is silently skipped when `allowedOrigins` is empty — LOW
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:343-346`
**Attack**: `if (this.allowedOrigins.length > 0 && origin && !this.allowedOrigins.includes(origin))` — an empty `allowedOrigins` config (the default) skips the origin check entirely. Combined with `hostMode='network'` binding on `0.0.0.0`, any origin-bearing browser request from the LAN can call the API if it has a valid bearer/session. The API is JSON-only (no HTML-rendering surface) so XSS/CSRF are bounded, but the current design implicitly trusts "any origin" rather than defaulting to "same-origin or configured".
**Impact**: When the daemon is used with a browser companion app on the LAN, a malicious intranet page could attempt requests with the session cookie (`SameSite=Lax` blocks cross-site CSRF — so real-world impact is muted). Not a direct vuln; rather a defense-in-depth gap.
**Recommendation**: When `hostMode` is `network` or `custom`, require `allowedOrigins` to be set explicitly (refuse to start, or log a loud warning). Optionally emit `Vary: Origin` + `Access-Control-Allow-Origin` only for configured origins.

### [SEC-08] HTTP-hook runner fetches user-controlled URL with no SSRF class filter — LOW
**File(s)**: `packages/sdk/src/_internal/platform/hooks/runners/http.ts:22-30`, webhook notifier `packages/sdk/src/_internal/platform/integrations/webhooks.ts:165-175`
**Attack**: The hook URL comes from `hook.url` (hooks.json, authored by the operator). `fetch(url, …)` has no scheme/host filter — an operator that loads an attacker-controlled `hooks.json` (or a plugin/MCP config that sets a webhook URL) can cause the daemon to fetch `http://169.254.169.254/…` (cloud metadata), `http://127.0.0.1:…` (loopback internal services), or `file://` schemes that Bun's fetch may partially honor.
**Impact**: Cloud-metadata SSRF or internal-service probing from the daemon, but only when `hooks.json` is attacker-controlled — `hooks.json` is already a documented trust boundary (see `hooks/runners/command.ts:1-14`). The bar is still lower than executing arbitrary shell (which the command runner also exposes).
**Recommendation**: Apply `classifyHostTrustTier` (already exists in `tools/fetch/trust-tiers.ts`) to reject loopback + link-local + cloud-metadata IPs unless the hook explicitly opts into `allowInternal: true`. Parallel to the existing outbound-TLS plumbing.

### [SEC-09] Shared `matchesSharedToken` length-prefix check leaks token length via timing — LOW
**File(s)**: `packages/sdk/src/_internal/platform/security/http-auth.ts:69-72`
**Attack**: `if (token.length !== sharedToken.length) return false;` short-circuits *before* `timingSafeEqual`. A remote attacker measuring response latency on `/login`-adjacent or bearer-protected endpoints can distinguish "wrong length" (immediate-return) from "right length, wrong bytes" (scrypt or crypto-cost). Token length is typically fixed (`randomBytes(24).toString('base64url')` yields a constant-length string), so the information leak is near-zero — but the defense-in-depth pattern is to pad to a fixed length and always invoke `timingSafeEqual`.
**Impact**: Negligible for the current fixed-length token scheme. Becomes material only if the daemon ever accepts variable-length shared tokens.
**Recommendation**: Normalize both sides to a fixed length before comparison, or compare a hash of each. Low priority.

### [SEC-10] `Math.random()` used for id generation in several call sites — LOW (INFO)
**File(s)**: `packages/sdk/src/_internal/platform/runtime/sandbox/session-registry.ts:16`, `runtime/settings/control-plane-store.ts:236`, `runtime/compaction/strategies/boundary-commit.ts:29`, `tools/exec/runtime.ts:126`, `integrations/delivery.ts:242`, `transport-http/http-core.ts:52` (generating random bytes via `Math.random`!)
**Attack**: `transport-http/http-core.ts:51-52` uses `bytes[i] = Math.floor(Math.random() * 256)` — if this feeds any security-relevant field (nonce, request id used for correlation-auth) the predictability matters. The other `Math.random().toString(36).slice(2, 8)` call sites generate 6-char ids used for user-visible correlation only (sandbox session names, delivery ids) — not security-critical. Worth confirming the `http-core.ts` usage is purely a client-side request id and not a bearer/nonce.
**Impact**: Likely INFO — none of these feed authentication. But the `Math.random` `bytes` generator in `http-core.ts` is a code smell; `crypto.randomBytes` or `crypto.getRandomValues` is the correct tool and costs nothing.
**Recommendation**: Replace `Math.random`-based byte generators with `crypto.getRandomValues(new Uint8Array(n))`. For string ids where unpredictability matters, use `randomUUID()` or `randomBytes(n).toString('hex')`.

### [SEC-11] Trigger-executor parses action string by whitespace-split — LOW
**File(s)**: `packages/sdk/src/_internal/platform/workflow/trigger-executor.ts:302-310`
**Attack**: `trigger.action.split(/\s+/)` is passed to `Bun.spawn(parts, …)`. Because this is `spawn` with an arg array (no shell), there is no shell-injection surface — good. However, any argument containing whitespace is silently fragmented into separate argv entries. An operator-configured trigger action of `echo "hello world"` is split into `['echo', '"hello', 'world"']`, which is not what the author expected and may cause silent misbehavior (or, combined with a tool that reads argv for secrets, unexpected exposure).
**Impact**: Correctness footgun, not a vuln. No shell injection.
**Recommendation**: Document the whitespace-split limitation loudly in the trigger config schema, or swap to a proper shell-quoting parser (e.g. shellwords).

### [SEC-12] `writeDaemonSetting` writes settings file without mode restriction — INFO
**File(s)**: `packages/sdk/src/_internal/platform/workspace/daemon-home.ts:213-227`
**Attack**: `writeFileSync(tmpPath, JSON.stringify(…), 'utf-8')` — no mode. `daemon-settings.json` is described in comments as identity state alongside `auth-users.json` and `operator-tokens.json`. Current contents are non-secret (binding mode, etc.) per the schema, but the path is treated as identity-state-grade in comments; the file permissions should match the invariant.
**Impact**: Informational — today's contents do not expose secrets, but a future addition might.
**Recommendation**: Align with `writeOperatorTokenFile` — `{ mode: 0o600 }` + post-rename `chmodSync(…, 0o600)`.

## Areas verified clean (no findings)

- Operator-token file (`companion-token.ts:82-83`, `daemon-home.ts:162-170`) correctly writes `{ mode: 0o600 }` + post-rename `chmodSync`. F3 regression is resolved.
- OTLP POST ingest (`telemetry-routes.ts:143-188`) correctly enforces `application/json` only (415 for protobuf), 4 MiB body cap (413), JSON-object shape check (400), and requires authentication before parsing. F7 concern is closed.
- HMAC signature verification in Slack / GitHub / webhook / trust plugin paths uses `timingSafeEqual` with length-prefix guard — constant-time compare is in place.
- Session cookies are `HttpOnly; SameSite=Lax; Path=/` with conditional `Secure` based on `trustProxy` — reasonable for a bearer-style local daemon.
- Spawn-token HMAC (`spawn-tokens.ts`) uses per-session `randomBytes(32)` secret, HMAC-SHA256 sign + `timingSafeEqual` verify, and revocable token registry — sound design.
- Prompt-injection surface was not deeply examined but no obvious user→system-prompt concatenation without a system/role boundary was found in the routes inspected.
- `__proto__` / prototype-pollution defenses present in `state/kv-state.ts` and `tools/state/index.ts` for user-keyed maps.
- Outbound `fetch` interceptor (`runtime/network/outbound.ts:239-261`) only lowers `rejectUnauthorized` for `isLocalHostname` URLs when the operator explicitly opted in via `allowInsecureLocalhost` — narrow and intentional.
