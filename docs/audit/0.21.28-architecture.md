# Architecture Audit — SDK 0.21.28 (commit 395fb32)

**Scope**: Full package layering, canonical/mirror integrity, facade/platform split, hexagonal boundaries, contract package purity, test architecture, error taxonomy, configuration schema, event domains.

**Methodology**: Static inspection via precision_engine tools + `bun run sync:check` (passes) + cross-reference against `.goodvibes/memory/` decisions/failures.

---

## Findings

### [ARCH-01] Platform HTTP layer has its own type surface uncovered by sync:check — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http/runtime-route-types.ts:47-204`, `packages/sdk/src/_internal/platform/daemon/http/router.ts`, `packages/sdk/src/_internal/platform/daemon/http/router-route-contexts.ts`, `packages/sdk/src/_internal/platform/daemon/http/provider-routes.ts`
**Issue**: `DaemonRuntimeRouteContext` is re-declared as `Omit<SdkDaemonRuntimeRouteContext, 'trySpawnAgent'>` and then **inlines** full duck-typed shapes for `sessionBroker` (lines 52-150), `agentManager` (151-154), `automationManager` (155-168), etc. This is the same fault pattern logged in `failures.json#0.21.24-platform-http-layer-missing-getSchedulerCapacity` — any addition to the canonical daemon-sdk `automationManager` shape (e.g. a new method) must be manually propagated to this local shape, and `sync:check` does NOT catch it (it only compares `packages/*/src/` against `packages/sdk/src/_internal/<subsystem>/`). The HTTP layer at `_internal/platform/daemon/http/**` is a THIRD surface outside sync coverage.
**Recommendation**: Either (a) replace the inlined shapes with direct imports from `../../../daemon.js` (single source of truth), or (b) extend `sync:check` with a `platform-http-drift` verifier that compares `_internal/platform/daemon/http/runtime-route-types.ts` field names against the canonical `DaemonRuntimeRouteContext`. Option (a) is cheaper; the `Omit<>` pattern already proves composition works.

### [ARCH-02] `daemon-sdk` redefines `RuntimeEventDomain` as `string` instead of importing from contracts — MEDIUM
**File(s)**: `packages/daemon-sdk/src/integration-route-types.ts:3`
**Issue**: `export type RuntimeEventDomain = string;` — the canonical union is defined in `packages/contracts/src/generated/runtime-event-domains.ts` with 29 literal members (session, turn, providers, …, workspace). daemon-sdk already depends on `@pellux/goodvibes-contracts` (per its package.json), so this wide alias is gratuitous type erosion. `createEventStream` in `IntegrationHelperServiceLike` accepts `readonly RuntimeEventDomain[]`, so callers can pass any string.
**Recommendation**: `import type { RuntimeEventDomain } from '@pellux/goodvibes-contracts';` and re-export it from daemon-sdk's index. This also eliminates the `DaemonRuntimeEventDomain` alias duplication exported from daemon-sdk's index.ts:43.

### [ARCH-03] Parallel error taxonomies — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/types/errors.ts` (AppError + 6 subclasses), `packages/sdk/src/_internal/platform/runtime/tasks/manager.ts` (3 classes), `packages/sdk/src/_internal/platform/runtime/permissions/{policy-loader,divergence-dashboard,simulation}.ts`, `packages/sdk/src/_internal/platform/runtime/ops/control-plane.ts`, `packages/sdk/src/_internal/platform/providers/provider-not-found-error.ts`, etc. (18 classes total across 9 files)
**Issue**: Platform-internal errors (`TaskTransitionError`, `ProviderError`, `PolicySignatureError`, `DivergenceGateError`, `OpsIllegalActionError`, …) extend `Error` or a local `AppError` base, not `GoodVibesSdkError` from `@pellux/goodvibes-errors`. They carry no `category`, `source`, `code`, `recoverable`, or `kind` fields. When these surface over HTTP they must be adapted to a `StructuredDaemonErrorBody`; the mapping logic is scattered rather than centralized through the errors package.
**Recommendation**: Migrate platform errors to extend `GoodVibesSdkError` with appropriate `category` (`'tool'`, `'bad_request'`, `'not_found'`, …) and `source` (`'runtime'`, `'security'`, …). Alternatively, introduce an internal `PlatformError` base in `packages/errors` so the taxonomy is a single tree.

### [ARCH-04] Oversized composition/schema modules violate SRP — LOW
**File(s)**: `packages/sdk/src/_internal/platform/daemon/facade.ts` (803 LOC), `packages/sdk/src/_internal/platform/daemon/http/router.ts` (637 LOC), `packages/sdk/src/_internal/platform/config/schema-types.ts` (773 LOC), `packages/sdk/src/_internal/platform/config/schema-domain-core.ts` (621 LOC), `packages/sdk/src/_internal/platform/daemon/facade-composition.ts` (570 LOC)
**Issue**: `facade.ts` (DaemonServer lifecycle + route dispatch + service composition) and `router.ts` (single router that maps paths to handler families) both carry many concerns. `schema-types.ts` mixes `ConfigKey` union, `ConfigValue<K>` conditional, and several domain-specific interfaces in one 773-line file — mutation at the three keep-in-sync sites (W6 in `failures.json`) is error-prone at this size.
**Recommendation**: `facade.ts` — split into `lifecycle.ts` (start/stop) and keep `facade.ts` for composition. `router.ts` — extract per-family mount files (already partially done with `router-route-contexts.ts`). `schema-types.ts` — split by domain (core / runtime / surfaces) matching the three `schema-domain-*.ts` siblings, keeping `ConfigKey` and `ConfigValue<K>` colocated with their domain source.

### [ARCH-05] `DaemonRuntimeRouteHandlerMap` duplicates handler list in two places — MEDIUM
**File(s)**: `packages/daemon-sdk/src/runtime-routes.ts` (Pick<> return type listing 30 handlers), `packages/sdk/src/_internal/platform/daemon/http/runtime-route-types.ts:208-241` (identical Pick<> as `DaemonRuntimeRouteHandlerMap`)
**Issue**: Same 30-key Pick<> appears in canonical daemon-sdk AND in the platform HTTP layer type file. sync:check does cover the canonical mirror at `_internal/daemon/runtime-routes.ts`, but the `_internal/platform/daemon/http/runtime-route-types.ts` duplicate is outside sync coverage and must be manually kept in sync. Same risk class as ARCH-01.
**Recommendation**: Import `DaemonRuntimeRouteHandlerMap` from `../../../daemon.js` rather than redeclaring it.

### [ARCH-06] Integration-test split is reasonable but handler-direct tests dominate — LOW
**File(s)**: `test/*.test.ts`
**Issue**: Grep finds 15 test files calling `dispatch*Routes(`/`create*RouteHandlers(` directly vs 6 using `DaemonHttpRouter.dispatchApiRoutes`. Handler-direct tests do not catch router-level wiring regressions (per `F-PROV-009-secrets-resolution-skipped-e2e-test` in resolved failures).
**Recommendation**: For every route family, require at least one router-level E2E test that exercises `Request → DaemonHttpRouter.dispatchApiRoutes → handler` end-to-end. Continue handler-direct tests for pure business-logic coverage, but gate architecture-critical wiring (auth, context threading, error adaptation) on E2E.

### [ARCH-07] `_internal/platform` sub-directories not covered by any drift/mirror guard — INFO
**File(s)**: `packages/sdk/src/_internal/platform/**`
**Issue**: Everything under `_internal/platform` is SDK-only code with no upstream canonical; it is legitimately outside sync:check. But this means tsconfig project references are the only architectural enforcement — there's no lint rule forbidding `_internal/platform` from importing the mirror (`_internal/daemon`) instead of the canonical (`packages/daemon-sdk`). Verified: all `_internal/platform/daemon/http/*.ts` imports of daemon routes use `../../../daemon.js` (the re-export barrel), which resolves to the mirror. That is correct for bundle layout but could mask drift if someone ever edited the mirror directly (W3 in failures.json).
**Recommendation**: Add a lint/CI check: files under `packages/*/src` (non-sdk) must not import from `_internal/`. Files under `_internal/platform` must route through the `daemon.js` / `contracts.js` / `errors.js` barrels, never directly into `_internal/<subsystem>/`.

### [ARCH-08] Facade/platform split is clean — INFO
**File(s)**: `packages/sdk/src/_internal/platform/daemon/facade.ts`, `facade-composition.ts`, `http/router.ts`
**Issue**: None — grep confirms ZERO `_internal/platform/**` files (outside `daemon/`) import facade/facade-composition. The dependency direction `facade → platform/*` is enforced by file layout. `http/` is imported by `facade` (4 files, 7 refs) but never vice-versa.
**Recommendation**: Keep it this way; document the rule in `docs/architecture.md`.

### [ARCH-09] `packages/contracts` is runtime-pure modulo zod — INFO
**File(s)**: `packages/contracts/src/**`
**Issue**: Contracts depends only on `zod/v4` (validation runtime, expected) and `node:url` (only in the `./node` export entrypoint). No other runtime deps, no transport/daemon imports. Clean.
**Recommendation**: Keep the `./node` subpath isolated; document that the default `.` export is runtime-neutral.

### [ARCH-10] Public/internal API separation is well-enforced at the package-export level — INFO
**File(s)**: `packages/sdk/package.json` exports map, `packages/sdk/src/index.ts`
**Issue**: `@pellux/goodvibes-sdk` exposes `.`, `./auth`, `./contracts`, `./daemon`, `./operator`, `./peer`, `./transport-*`, `./browser`, `./web`, `./react-native`, `./expo`, and deliberately `./platform/*` as subpath exports. The `./platform/*` subpath IS the escape hatch for daemon binaries and TUI embedders that need internals; 24 files in the workspace use it. This is intentional (per `decisions.json#standalone-route-extension-pattern`) but the naming `_internal/platform` hints these are not stable — which they aren't.
**Recommendation**: Rename `./platform/*` exports to `./unstable/*` or add a prominent “unstable — ABI may change in any release” marker in `docs/packages.md` so downstream TUI authors do not mistake them for public API.

### [ARCH-11] daemon-sdk under-utilizes its `@pellux/goodvibes-errors` dependency — LOW
**File(s)**: `packages/daemon-sdk/src/error-response.ts`, `packages/daemon-sdk/src/**`
**Issue**: Only 2 files in daemon-sdk import `@pellux/goodvibes-errors`. Given daemon-sdk is the authoritative error-response builder (`buildErrorResponseBody`, `jsonErrorResponse`, `summarizeErrorForRecord`), more of its internal error handling should route through the errors-package types (`StructuredDaemonErrorBody`, `DaemonErrorCategory`, `DaemonErrorSource`).
**Recommendation**: Audit daemon-sdk error surfaces and ensure every `jsonErrorResponse` call site uses a `DaemonErrorCategory` constant from the errors package rather than string literals.

## Closing Notes

The 0.21.28 architecture is in good shape overall. The `contracts`/`errors`/`daemon-sdk`/transport packages form a clean layered stack with no circular deps, and `sync:check` keeps the 9 publish mirrors intact. The one lingering **structural risk** is the platform HTTP layer's locally-inlined type shapes (ARCH-01/ARCH-05), which has already caused one CI orphan (`0.21.24-platform-http-layer-missing-getSchedulerCapacity`). Collapsing those shapes onto imports from `daemon.js` would close the gap without requiring a new sync verifier.
