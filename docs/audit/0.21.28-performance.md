# SDK 0.21.28 — Performance & Memory-Leak Audit

**Commit**: `395fb32` (release: SDK 0.21.28)
**Scope**: `packages/sdk/src/_internal/platform/**` (runtime, control-plane, core, daemon, scheduler, telemetry, mcp)
**Method**: static read — grep + targeted reads. No runtime profiling performed.

Citations are `file:line`. Paths are relative to the SDK repo root.

### [PERF-01] SessionBroker never evicts closed sessions or their buckets — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/control-plane/session-broker.ts:55-56`, `:1022-1051` (`_gcSweep`)
**Issue**: `_gcSweep()` runs every 60s and transitions idle sessions to `status: 'closed'` via `this.sessions.set(sessionId, closed)`. It never calls `this.sessions.delete`, `this.messages.delete`, or `this.inputs.delete`. Across a long-running daemon, the three `Map`s accumulate every session ever created plus the full trimmed message/input buckets for each. `listSessions()` (line 264) and `findPreferredSession()` (line 273) traverse all closed sessions on every call (`sortSessions(this.sessions.values())`). The `MAX_PERSISTED_MESSAGES = 2_000` cap (line 41) bounds each bucket individually but the outer map grows linearly with session count. Inputs have no per-bucket cap at all (only `sortInputs` is called in `setInput` at :899).
**Reproduction**: start daemon, create N sessions, let them idle past `_idleLongMs`. Inspect `broker.sessions.size` — will be N. Heap retention: O(N × MAX_PERSISTED_MESSAGES).
**Recommendation**: add a hard retention window (e.g. 7d after `closedAt`) inside `_gcSweep` that actually deletes `sessions`/`messages`/`inputs` entries. Add a `MAX_PERSISTED_INPUTS` for symmetry in `setInput`.

### [PERF-02] RateLimiter LRU uses O(n) accessOrder on every HTTP request — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/daemon/http-listener.ts:49-103`
**Issue**: `RateLimiter.check()` does `this.accessOrder.indexOf(ip)` (O(n)) and `splice` (O(n) after the cut point) on every single HTTP request. With `RATE_MAX_ENTRIES = 10_000`, each request does up to 20k array ops just for LRU bookkeeping, before rate decisions. `_sweep()` also does an O(n) `accessOrder.indexOf` + `splice` per evicted entry — potentially O(n²) worst-case every sweep.
**Reproduction**: benchmark a webhook burst with varied source IPs; compare p99 under 10 vs 10k IP diversity.
**Recommendation**: replace `accessOrder: string[]` with a `Map<ip, lastSeen>` (Map iteration order IS insertion order in JS) — use `delete` + `set` for promotion (both O(1)). Or adopt a standard LRU library.

### [PERF-03] Scheduler pushHistory is O(history.length) per task run — HIGH
**File(s)**: `packages/sdk/src/_internal/platform/scheduler/scheduler.ts:661-677`, `:683-688`
**Issue**: `pushHistory` does `history.filter(...)` twice (once to count the per-task entries, once to produce a trimmed copy) on every single task run. `MAX_HISTORY_PER_TASK = 5` bounds per task, but global `history.length` grows as 5 × (distinct taskIds), and the filter is global-scan. Additionally, `save()` serializes `history: this.history` (entire array) to disk on every run. Over thousands of tasks this becomes noticeable.
**Reproduction**: create 1k scheduled tasks, fire them sequentially, time `_onTaskFired` latency.
**Recommendation**: index history by taskId (`Map<taskId, RunRecord[]>`), giving O(1) push + O(1) trim. Merge on `save()`. Additionally: `save()` on every run is wasteful — batch-debounce the persist (e.g. 1s coalescing window).

### [PERF-04] Telemetry SSE heartbeat leaks on `cancel()` path — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/telemetry/api.ts:396-415`
**Issue**: The SSE stream installs `setInterval(heartbeat, 15_000)`. The `abort` listener calls `clearInterval(heartbeat)` but the `cancel` callback (stream-closed path, line 412) only calls `unsub()` — it does not `clearInterval(heartbeat)`. If the consumer cancels the stream without the request being aborted, the interval runs forever, enqueueing into a closed controller (which throws) and keeps the process alive. Also missing `.unref()` — see PERF-07.
**Reproduction**: open the SSE endpoint via a reader whose `.cancel()` fires without `AbortController.abort()`; the interval continues to fire.
**Recommendation**: move `clearInterval(heartbeat)` into a shared teardown closure called from both `abort` and `cancel` (see `runtime/integration/helpers.ts:617-628` which does this correctly).

### [PERF-05] Gateway SSE heartbeat also without `.unref()` — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/control-plane/gateway.ts:628-700`
**Issue**: The Gateway opens a `ReadableStream` with `setInterval(heartbeat, 15_000)` at line 655. `teardown()` does `clearInterval(heartbeat)` (good), but the interval is never `.unref()`'d. If an SSE client hangs without triggering teardown (e.g. socket half-closed), the interval prevents the Node/Bun process from cleanly exiting on SIGTERM. Separately, `controller.enqueue(...)` on the bus onDomain callback has **no backpressure check** — a slow/stalled consumer buffers unbounded serialized envelopes inside the ReadableStream's internal queue until memory pressure kicks in.
**Reproduction**: connect an SSE client that reads at 1 event/sec while the runtime emits at 1k/sec. Observe RSS growth.
**Recommendation**: `heartbeat.unref?.()`. For backpressure: check `controller.desiredSize` before each enqueue; if ≤ 0, drop or coalesce (surface a `control-plane.backpressure` diagnostic).

### [PERF-07] Multiple long-lived setInterval timers missing `.unref()` — MEDIUM
**File(s)**:
- `runtime/telemetry/api.ts:396` (SSE heartbeat)
- `control-plane/gateway.ts:655` (SSE heartbeat)
- `runtime/integration/helpers.ts:614` (SSE heartbeat)
- `daemon/http-listener.ts:58` (rate-limiter sweep)
- `runtime/bootstrap-runtime-events.ts:248` (agent-status broadcast)
- `runtime/perf/slo-collector.ts:129` (pending sweep)
- `core/orchestrator.ts:467` (anim frame)
- `runtime/mcp/manager.ts:472` (reconnect timer)
- `control-plane/session-broker.ts:257` — **correctly calls `.unref()`** (reference)
- `companion/companion-chat-manager.ts:186-191` — **correctly calls `.unref()`** (reference)

**Issue**: Timers without `.unref()` keep the Bun/Node event loop alive, blocking clean process exit on SIGTERM until an explicit `clearInterval` is reached. The codebase already demonstrates awareness of the pattern (two call sites do it correctly). The remaining sites should follow.
**Recommendation**: append `(iv as unknown as { unref?: () => void }).unref?.();` after each `setInterval` that is not essential for keeping the process alive.

### [PERF-08] TelemetryApiService uses full-array copy on every list query — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/telemetry/api.ts:677`, `:704`
**Issue**: `applyRecordFilter` does `[...records].filter(...)` — allocates a shallow copy of up to `eventLimit` (default 500) records per call. `listSpans` similarly: `[...this.spans].filter(...)` at line 704. If telemetry UIs poll every few seconds, and span/record arrays approach the default limit, this is 500-entry allocations per poll per client per call-path. Filter is already non-mutating; the spread is redundant.
**Reproduction**: open the telemetry UI, profile; count allocations in `applyRecordFilter`.
**Recommendation**: drop the spread — `records.filter(...)` already returns a new array. Same at :704.

### [PERF-09] DistributedRuntime audit array has no in-memory cap — MEDIUM
**File(s)**: `packages/sdk/src/_internal/platform/runtime/remote/distributed-runtime-store.ts:62-64` (load), `:14` (audit declaration via Array)
**Issue**: `state.audit.push(normalized)` grows unbounded in memory. `pruneAndPersistDistributedRuntime` (:67) runs on load only — subsequent audits between loads accumulate without a cap check. Need to verify whether all push paths go through a bounded `appendAudit`-style helper; the direct `.push` on line 64 is suspicious.
**Reproduction**: drive a stream of audit events without triggering a snapshot persist; observe `state.audit.length`.
**Recommendation**: call `pruneAndPersistDistributedRuntime` (or an in-memory `pruneAudit`) at each push, or use a shared `appendBounded` helper (already present in `runtime/telemetry/api-helpers.ts:84`).

### [PERF-10] RuntimeEventBus warns but does not enforce MAX_LISTENERS — LOW
**File(s)**: `packages/sdk/src/_internal/platform/runtime/events/index.ts:55-127`
**Issue**: `MAX_LISTENERS = 100` triggers a `logger.warn` only. Consumers with leak bugs cascade into full memory retention because the bus holds strong references in `_listeners` Sets. Warning is printed once per threshold cross but the listener stays installed forever.
**Recommendation**: add a telemetry span emission in addition to the log (already partially visible); consider a hard ceiling + refuse-on-overflow in dev mode.

### [PERF-11] Sync fs operations in what may be long-running code paths — LOW
**File(s)** (selected high-value hits; 24 files total):
- `artifacts/store.ts:299-303` — `existsSync/statSync` during artifact resolution
- `workspace/daemon-home.ts` multiple sites — daemon-home resolution on each call
- `daemon/service-manager.ts:282,301,382,436` — `existsSync/statSync` during service lifecycle
- `watchers/store.ts:33,55`
- `discovery/mcp-scanner.ts:95,102,144,148`

**Issue**: All are sync fs in arguably cold-path admin code (daemon boot, artifact writes) — not on the per-request hot path. Still, on daemons with slow filesystems (network mounts, WSL), sync calls block the single event loop. Some — e.g. `daemon-home.ts` — are likely called repeatedly.
**Recommendation**: spot-audit each site's call frequency. Convert any that fire during request handling or event emission to async `fs/promises`.

### [PERF-13] session-broker.ts — `sortInputs` on every input write — LOW
**File(s)**: `packages/sdk/src/_internal/platform/control-plane/session-broker.ts:899`
**Issue**: Each `setInput` re-sorts the entire inputs bucket. Inputs are naturally time-ordered; a sort is unnecessary except on reconciliation. For a session with many inputs, this is O(n log n) per write.
**Recommendation**: insert in sorted position (binary search) or only sort on explicit load/reconcile.

**Issue**: `this.entries.push` has no cap. Typical growth is slow (one entry per compaction). Included for completeness; in practice unlikely to dominate.

### [PERF-15] Cache-return shallow copies everywhere — INFO
Several classes return `[...this._internal]` defensive copies on every read (18 sites found: `providers/registry.ts:442,454,805`, `integrations/delivery.ts:272,304`, `permissions/policy-registry.ts:450`, etc.). Individually cheap; collectively they flood the minor heap in hot code. Prefer returning `readonly` views or iterators unless mutation isolation is explicitly required.

## Top 3 concerns

1. **PERF-01** — SessionBroker never deletes closed sessions. On any long-running daemon, `sessions`/`messages`/`inputs` Maps grow without bound. All `listSessions`/`findPreferredSession` calls degrade linearly.
2. **PERF-02** — RateLimiter LRU is O(n) per HTTP request with a 10k cap → up to 20k array operations per call, before the rate check runs.
3. **PERF-03** — Scheduler `pushHistory` does two global O(n) filters per task run plus full-array persist, compounding as distinct task count grows.
