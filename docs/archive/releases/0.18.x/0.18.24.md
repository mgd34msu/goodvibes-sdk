# 0.18.24

`0.18.24` fixes the next SDK host-boundary leak in the ecosystem/catalog surface.

The bug was straightforward:

- the reusable SDK ecosystem helpers were still hardcoding catalog and receipt files under `.goodvibes/ecosystem/...`
- `goodvibes-tui` keeps its host-owned curated catalog and receipt policy under `.goodvibes/tui/ecosystem/...`
- once the TUI command surfaces were rewired onto the published SDK package, plugin, skills, and marketplace commands stopped finding the TUI-owned catalog files even though the data was present

This release removes that hardcoded storage assumption from the SDK and pushes the catalog/receipt placement back to the host boundary where it belongs.

## Why This Release Exists

The SDK and TUI have different responsibilities here.

The SDK should define:

- how curated ecosystem catalogs are loaded
- how entries are reviewed, installed, updated, rolled back, and uninstalled
- how install receipts and catalog bundles are serialized

The host should define:

- where curated catalog JSON lives
- where install receipts are stored
- which surface-specific config tree owns those files

The previous SDK release still violated that boundary by baking `.goodvibes/ecosystem/...` into the reusable ecosystem module.

That was wrong for `goodvibes-tui`, and it would also be wrong for any other host that wants curated catalogs and receipts under its own surface-specific config layout.

## What Changed

### Host-Configurable Ecosystem Catalog Roots

`platform/runtime/ecosystem/catalog` now accepts host-owned catalog roots through `EcosystemCatalogPathOptions`:

- `projectCatalogRoot`
- `userCatalogRoot`

If a host does not provide them, the SDK keeps the previous default behavior:

- project catalog root defaults to `<cwd>/.goodvibes/ecosystem`
- user catalog root defaults to `<homeDir>/.goodvibes/ecosystem`

That preserves backward compatibility for generic hosts while allowing TUI and other surfaces to put curated catalog files where they actually belong.

### Host-Configurable Receipt Roots Without Breaking Install Destinations

The ecosystem module also now supports host-owned install-base overrides:

- `projectInstallRoot`
- `userInstallRoot`

For the TUI cutover, the important detail is that the host only overrides the catalog/receipt side.

Installed artifacts still land in the shared GoodVibes runtime locations:

- project plugins: `.goodvibes/plugins/...`
- project skills: `.goodvibes/skills/...`
- project hook packs: `.goodvibes/hooks/packs/...`
- project policy packs: `.goodvibes/policies/packs/...`

That means the SDK now supports the correct split:

- curated catalogs and receipts under `.goodvibes/tui/ecosystem/...`
- installed runtime artifacts under shared `.goodvibes/...` roots

### A Locked Regression Test

This release adds an SDK regression test that covers the exact boundary that broke the TUI:

- curated plugin catalog file under `.goodvibes/tui/ecosystem/plugins.json`
- install receipt under `.goodvibes/tui/ecosystem/installed/...`
- installed plugin files under `.goodvibes/plugins/...`

That test now proves the SDK can honor host-owned catalog roots without rewriting the shared runtime install destinations.

## Downstream Impact

This patch is required for the TUI SDK cutover because it restores the intended host model:

- `goodvibes-tui` config/storage policy remains TUI-owned
- the SDK provides the reusable catalog/install/update/review logic
- the TUI can consume the published SDK package directly without carrying a local ecosystem implementation just to keep `.goodvibes/tui/...` paths working

The practical surfaces fixed by this release are the SDK-backed:

- plugin command catalog browse/review/install flows
- skills command curated catalog flows
- marketplace overview/browse/review/install flows
- ecosystem catalog bundle/receipt flows used by the TUI runtime and tests

## Validation

This release was validated with:

- `bun x tsc -b packages/sdk --force --pretty false`
- `bun test test/ecosystem-catalog-paths.test.ts`
- `bun run validate`

The added regression test confirms the host-owned catalog root behavior directly against the built SDK package surface.
