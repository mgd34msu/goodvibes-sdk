# 0.18.40 — CI recovery: version.ts format + sync script hardening

**Released**: 2026-04-16
**Type**: Patch (CI fix; no functional changes to shipped SDK code)

## Summary

0.18.39 shipped correctly but its CI run failed because `packages/sdk/src/_internal/platform/version.ts` was written as:

```ts
let version = "0.18.39";;
```

(double-quoted string + duplicate semicolon). `test/version-sync.test.ts` matches on `/let version = '([^']+)'/` (single quotes) and thus received `null`, failing validate and blocking the npm publish.

This release:

1. Restores `version.ts` to the canonical `let version = '0.18.40';` format
2. Hardens `scripts/sync-version-fallback.ts` so its regex tolerates either quote style plus stray duplicate semicolons. Future accidental edits to the wrong format get self-healed on the next `bun run sync:version` invocation

All 0.18.39 code correctness fixes carry forward unchanged in 0.18.40.

## What changed

- `packages/sdk/src/_internal/platform/version.ts` — quote style corrected
- `scripts/sync-version-fallback.ts` — regex broadened to `['"][^'"]*['"];+` so both quote styles and one-or-more trailing semicolons are normalized to `let version = 'X.Y.Z';`
- All workspace `package.json` versions bumped 0.18.39 → 0.18.40
- `CHANGELOG.md`

## Migration

None.

## Why not amend 0.18.39?

The `v0.18.39` tag was pushed to remote. Force-pushing tags or rewriting history on main is prohibited by policy, so the cleanest path is to ship 0.18.40 with the fix and leave 0.18.39 as a "broken release" marker in the tag history (no npm artifact exists at that version).
