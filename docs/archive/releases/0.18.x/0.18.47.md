# 0.18.47 — stopReason canonical vocab + authToken normalization

**Released**: 2026-04-16
**Type**: Additive (type-narrowing on `stopReason`)

## Summary

Previously, the `stopReason` field on `ChatResponse` carried whatever string each
provider returned (`'end_turn'`, `'stop'`, `'length'`, etc.), forcing consumers to branch
on provider-specific vocabulary. This release normalizes `stopReason` to a fixed union of
seven canonical values and narrows its type from `'end' | 'tool_use' | 'max_tokens' | 'error'`
to `ChatStopReason`. The raw provider string is preserved at `providerStopReason` for
consumers that need it. A new `normalizeAuthToken()` helper is also exported for
consumers that want a single canonical path for normalizing `string | object | function`
auth-token inputs into a reusable resolver.

## Changes

### `ChatResponse.stopReason` canonical vocab

`stopReason` is now typed as the new `ChatStopReason` union:

```ts
type ChatStopReason =
  | 'completed'       // Natural end of generation (was 'end')
  | 'max_tokens'      // Output token limit reached
  | 'tool_call'       // Model requested tool invocation (was 'tool_use')
  | 'stop_sequence'   // Matched an explicit stop sequence
  | 'content_filter'  // Provider content filter triggered
  | 'error'           // Generation aborted due to error
  | 'unknown';        // Fallback for unmapped provider values
```

The previous raw provider value is preserved at `providerStopReason?: string` for
consumers that need the original signal.

All 10 provider adapters now call a canonical mapper from the new
`providers/stop-reason-maps.ts` module at their finish-reason emission site. No inline
ternaries or local duplicate map functions remain.

### `normalizeAuthToken()` helper

New export from `@pellux/goodvibes-sdk/transport-http`:

```ts
import { normalizeAuthToken } from '@pellux/goodvibes-sdk/transport-http';

// Types:
export type AuthTokenResolver = () => MaybePromise<string | null | undefined>;
export type AuthTokenInput =
  | string
  | { readonly token: string }
  | AuthTokenResolver
  | (() => string | null | undefined | Promise<string | null | undefined>)
  | undefined;

export function normalizeAuthToken(input: AuthTokenInput): AuthTokenResolver;
```

`normalizeAuthToken` **normalizes** any supported input form into a canonical
`AuthTokenResolver` function. It does not resolve the token itself — the caller invokes
the returned resolver:

```ts
const resolver = normalizeAuthToken(input); // returns AuthTokenResolver
const token = await resolver();             // resolves the token
```

Consumers that already pass the right shape to their provider do not need to change
anything. The helper is opt-in for those who want a single resolver.

## Breaking behavior

`ChatResponse.stopReason` narrows from `'end' | 'tool_use' | 'max_tokens' | 'error'` to
the `ChatStopReason` union. TypeScript will surface an error for any `switch` branch or
comparison that uses a value not present in `ChatStopReason`. Update to the canonical
values listed above. Every prior raw provider value maps to exactly one canonical value,
so runtime behavior is unchanged.
