# 0.18.20

`0.18.20` fixes a real SDK runtime regression in the REPL tool's Python path.

The previous package still tried to create an ephemeral virtual environment before every Python evaluation. That is the wrong default for a reusable SDK runtime surface, and it broke the TUI immediately in environments where `python3` exists but `python3 -m venv` fails because `ensurepip` or distro venv packaging is unavailable.

## Why This Release Exists

The TUI cutover moved the REPL tool onto the canonical package export:

- `@pellux/goodvibes-sdk/platform/tools/repl`

That exposed an SDK-owned bug:

- the Python REPL branch created a throwaway virtualenv under the workspace
- that required `python3 -m venv` to succeed on every eval run
- on this host, `python3` was present and usable, but `venv` failed while trying to bootstrap `ensurepip`

That meant the published SDK still broke a real host workflow even though the host already had a working Python interpreter.

## What Changed

### Python REPL Runtime

The SDK no longer tries to provision an ephemeral virtualenv for Python evals.

Instead, the REPL tool now:

- runs `python3` directly inside the selected sandbox session
- keeps the same isolated `-I -S` execution flags
- executes from the host-provided workspace root instead of a throwaway venv directory

### Boundary Correction

This is the correct boundary for the SDK:

- the host decides whether `python3` is available
- the sandbox/runtime layer decides how to execute it
- the REPL tool should not silently require extra Python environment bootstrapping just to evaluate a simple expression

The SDK still defines the runtime behavior. It no longer smuggles in an extra packaging assumption about Python's `venv` support.

## TUI Impact

This patch unblocks the TUI cutover by fixing the last targeted test failure in the migrated REPL/sandbox surface:

- `src/test/tools/repl.test.ts`

After this release, the TUI can consume the published SDK REPL tool without carrying a local Python REPL implementation or tripping over host-specific `venv` packaging.
