# Release 0.18.41

**Date**: 2026-04-16
**Type**: Reliability + Performance

## Summary

Three SDK reliability and performance fixes targeting exec retry behaviour, ProcessManager background process lifecycle, and Orchestrator abort cleanup.

## Changes

### Îł1 â€” exec retry: full jitter + retryable error classification

**Files**: `packages/sdk/src/_internal/platform/tools/exec/runtime.ts`, `packages/sdk/src/_internal/platform/tools/exec/schema.ts`

**Problem**: The `computeRetryDelay` function used exponential backoff without jitter, causing thundering-herd storms when multiple callers hit a flaky dependency simultaneously. `runWithRetry` retried all failures indiscriminately, including terminal errors like missing binaries and permission denials.

**Fix**:
- `computeRetryDelay` now uses full jitter: `Math.random() * min(base * 2^attempt, maxDelay)`. The cap is configurable via `retry.max_delay_ms` (default 30000ms).
- `isRetryableExecResult` classifies exec failures. Terminal errors (ENOENT, EACCES, `command not found`, `Permission denied`, syntax errors, `No such file or directory`) stop the retry loop immediately. Network errors (ECONNRESET, ENOTFOUND, ETIMEDOUT), lock/busy (EBUSY, ECONNREFUSED), and OOM (ENOMEM) are retried per the `retry.on` allowlist.
- `retry.on` field added to `ExecRetry` schema and interface, allowing callers to restrict which error categories trigger retry.
- `isRetryableExecResult` is now exported for testability.

### Îł2 â€” ProcessManager.spawn: timeout + SIGKILL deadline + error surfacing

**Files**: `packages/sdk/src/_internal/platform/tools/shared/process-manager.ts`, `packages/sdk/src/_internal/platform/tools/exec/runtime.ts`

**Problem**: `ProcessManager.spawn()` was fire-and-forget with no timeout. Hanging background processes had no recourse. ENOENT/EACCES from `Bun.spawn` were silently swallowed, leaving callers unable to distinguish spawn failures from successful background spawns.

**Fix**:
- `spawn()` is now `async` and accepts `SpawnOptions` with `timeout_ms` (default 60000) and `sigterm_grace_ms` (default 5000).
- A timeout watchdog sends SIGTERM after `timeout_ms`, waits `sigterm_grace_ms`, then sends SIGKILL.
- `BackgroundProcess.killDeadline` records when SIGKILL was scheduled (ms epoch), distinguishing timeout-kills from user-initiated stops in logs and observability.
- ENOENT/EACCES from `Bun.spawn` now reject the returned promise instead of silently creating an orphaned tracking entry.
- `SpawnOptions` interface exported for external use.

### Îł3 â€” Orchestrator.abort() clears animInterval

**Files**: `packages/sdk/src/_internal/platform/core/orchestrator.ts`

**Problem**: `abort()` killed the in-flight LLM stream and cleared `autoSpawnTimeout`, but left `animInterval` (the thinking-animation `setInterval`) running. This leaked a timer that kept the Node event loop alive and prevented clean shutdown when `abort()` was called from outside the turn loop.

**Fix**:
- `abort()` now calls `clearInterval(this.animInterval)` and sets `animInterval = null`.
- `isThinking` is set to `false` in `abort()` so UI state is consistent regardless of whether `stopThinking()` is reached in the `finally` block.
- The `dispose()` path already cleared the interval; `abort()` now mirrors it for the mid-turn-abort case.
