# 0.18.23

`0.18.23` fixes another SDK host-boundary leak by removing the concrete SDK `ConfigManager` class requirement from the reusable runtime network helpers.

## Why This Release Exists

The previous SDK package still typed the reusable inbound and outbound TLS/network helpers against the SDK's own `ConfigManager` class identity.

That was the wrong boundary.

A host should only need to provide:

- configuration reads for the specific network keys the helper needs
- the resolved control-plane config root for certificate path derivation

It should not need to instantiate the SDK's concrete config manager class just to use reusable network helpers.

## What Changed

### Host-Neutral Network Config Interfaces

The SDK network layer now accepts small structural interfaces instead of the concrete SDK config class:

- `platform/runtime/network/shared`
  - `NetworkRootConfig`
- `platform/runtime/network/inbound`
  - `InboundTlsConfigReader`
- `platform/runtime/network/outbound`
  - `OutboundTlsConfigReader`

These interfaces describe only the methods the network helpers actually need.

### Reusable TLS Helpers

The following helpers now work with host-owned config readers instead of SDK class identity:

- `getGoodVibesRootDir`
- `getDefaultCertDirectory`
- `getDefaultInboundCertPaths`
- `resolvePathFromGoodVibesRoot`
- `inspectInboundTls`
- `resolveInboundTlsContext`
- `inspectOutboundTls`
- `applyOutboundTlsToFetchInit`
- `createNetworkFetch`
- `GlobalNetworkTransportInstaller`

### Downstream Impact

This patch is required for the TUI migration because it removes another place where the SDK was still forcing the host to share a concrete implementation class instead of a reusable contract.

That means the TUI can import the SDK network helpers directly while keeping:

- `.goodvibes/tui/...` storage policy
- TUI-owned config manager construction
- TUI-owned host composition
