# 0.18.21

`0.18.21` fixes the replay/export path boundary so the SDK no longer hardcodes `/tmp` as the only non-project export root.

## Why This Release Exists

The TUI release-path validation now runs every test file under repo-local temp roots instead of the shared OS temp filesystem.

That exposed a real SDK boundary bug:

- `DeterministicReplayEngine.export(...)` still only allowed paths under the project root or literal `/tmp`
- that assumption is wrong for a reusable SDK because the host controls the active temp root
- once the TUI runner moved to repo-local temp roots, replay export validation started failing even though the export path was valid for the host

## What Changed

### Replay Export Boundary

The replay engine now accepts export paths under:

- the loaded project root
- the active temp root returned by `tmpdir()`

This keeps the safety boundary in place while removing the Unix-only `/tmp` assumption from the reusable package surface.

### Public Guidance Cleanup

The replay command help text now uses a project-local export example instead of telling consumers to write to `/tmp`.

That keeps the SDK guidance aligned with the actual host-owned configuration model:

- the SDK defines what the replay engine can export
- the host decides where temporary files live

## TUI Impact

This patch is specifically needed for the canonical TUI cutover release work:

- the TUI test runner now uses repo-local temp roots per test file
- replay export validation must therefore respect host temp configuration
- without this patch, the TUI still fails part of its release-path suite even though the rest of the SDK cutover is correct

## Validation

- `bun run validate`
