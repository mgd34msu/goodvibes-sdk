# 0.18.25

`0.18.25` fixes a startup-time native binding bug in the published SDK tool surface.

The bug was simple:

- the SDK tool modules for structural find and `ast_pattern` edit imported `@ast-grep/napi` at module top level
- hosts like `goodvibes-tui` register the full tool surface during startup
- that meant the compiled TUI binary tried to resolve the AST native binding before the user had asked for any AST feature at all

On a compiled TUI build, that was enough to crash `./dist/goodvibes` on launch with the `Cannot find native binding` error from `@ast-grep/napi`.

## What Changed

### Lazy AST Runtime Loading

The SDK now lazy-loads `@ast-grep/napi` only when an AST-backed feature is actually invoked:

- `platform/tools/find/structural`
- `platform/tools/edit/match`

That means:

- registering tools no longer loads the native AST runtime
- booting a host application no longer requires the AST native binding to be present immediately
- AST-backed features still work when invoked in environments where the binding is available

### `ast_pattern` Edit Path Updated

The edit runtime now awaits the lazy AST load for `ast_pattern` mode instead of relying on a synchronous top-level import.

Behavior stays aligned with the intended contract:

- if AST support is available, `ast_pattern` uses it
- if AST support is unavailable, the edit path falls back cleanly instead of crashing startup

### Regression Coverage

This release adds a regression test that locks the startup boundary in place by checking that the startup-sensitive tool modules no longer contain top-level `@ast-grep/napi` imports.

That protects the exact failure mode that broke the compiled TUI binary.

## Downstream Impact

This patch is required for hosts that consume the SDK as a package and register the shared tool surface during startup.

For `goodvibes-tui`, this is the fix that allows:

- `bun run build`
- `./dist/goodvibes`

to work again after the SDK cutover, without the compiled binary dying on `@ast-grep/napi` before any AST-backed command is used.

## Validation

This release was validated with:

- `bun x tsc -b packages/sdk --force --pretty false`
- `bun test test/lazy-native-imports.test.ts`
- `bun run validate`
- downstream confirmation that the compiled TUI binary only stops failing once the TUI is updated to the published patched SDK version
