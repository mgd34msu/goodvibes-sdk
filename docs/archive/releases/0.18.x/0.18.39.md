# 0.18.39 — Code review correctness fixes + cache invariant test coverage

**Released**: 2026-04-16
**Type**: Patch (correctness)

## Summary

Seven targeted fixes addressing correctness issues identified in the 0.18.38 code review, plus comprehensive unit test coverage for all 5 hot-path cache invariants introduced in 0.18.38.

## What changed

### I1: `setModelContextCap` customModels branch missing invalidation (blocker)

**File**: `packages/sdk/src/_internal/platform/providers/registry.ts`

- The `customModels` branch of `setModelContextCap` mutated `this.customModels[customIdx]` and returned without calling `_invalidateModelRegistry()`. Any caller who primed the `listModels()` cache before calling `setModelContextCap` on a custom model would continue to see the old context window value until the registry was next rebuilt for another reason.
- Fixed: `this._invalidateModelRegistry()` is now called before the `return` in the `customModels` branch, mirroring the existing pattern in the `discoveredModels` branch.

### I2: Unit tests for the 5 new cache paths

**File**: `test/cache-invariants.test.ts` (new)

Added 21 unit tests covering all 5 invariants:

- **(a)** `setModelContextCap` invalidation in `listModels()` for both the `customModels` path and the `discoveredModels` path
- **(b)** `registerRuntimeProvider` unregister callback invalidates the model registry cache
- **(c)** `recentEvents` ring buffer returns events newest-first for both `count < cap` (3-event ring) and `count >= cap` (600 events, 500-slot ring); `getSnapshot().totals.recentEvents` uses `_recentEventsCount` (O(1))
- **(d)** `_syncScheduled` coalesces 10 synchronous `rememberEvent` calls into exactly 1 `syncControlPlaneState` dispatch; `lastEventAt` reflects the most recent event's timestamp, not the first
- **(e)** `getMessagesForLLM` returns the same reference (`===`) on cache hits; returns a fresh reference after each of the 15 mutating methods: `addUserMessage`, `addAssistantMessage`, `addSystemMessage`, `addToolResults`, `removeMessagesAfter`, `startStreamingBlock+finalizeStreamingBlock`, `startStreamingBlock+updateStreamingBlock`, `replaceMessagesForLLM`, `resetAll`, `switchBranch`, `mergeBranch`, `fromJSON`, `undo`, `redo`, `markLastUserMessageCancelled`

### I3: `getSnapshot` O(1) fix for `recentEvents` count

**File**: `packages/sdk/src/_internal/platform/control-plane/gateway.ts`

- `getSnapshot()` was reading `this.recentEvents.length` which triggers the full O(n) ring accessor, allocating a new array just to read its length.
- Fixed: replaced with `this._recentEventsCount` which is O(1) and always accurate.

### I4: `lastEventAt` closure capture bug in `_scheduleControlPlaneSync`

**File**: `packages/sdk/src/_internal/platform/control-plane/gateway.ts`

- `_scheduleControlPlaneSync(lastEventAt)` was called with the timestamp of the first event in a synchronous burst, and that value was captured in the `setImmediate` closure. When N events arrived in a single tick, the dispatched `lastEventAt` was the timestamp of the first event, not the most recent.
- Fixed: added `private _lastEventAt = 0` field. `rememberEvent` updates it on every event. The `setImmediate` callback now reads `this._lastEventAt` instead of a closure-captured parameter. `_scheduleControlPlaneSync` signature updated (no longer accepts `lastEventAt` argument).

### I5: `registerDiscoveredProviders` invalidation timing

**File**: `packages/sdk/src/_internal/platform/providers/registry.ts`

- `_invalidateModelRegistry()` was called before the loop that populated `discoveredModels`. The pre-loop invalidation was safe only by accident (nothing inside the loop read the cache). Now the invalidation fires once after the loop, ensuring the rebuilt registry always reflects all newly registered providers and models.
- Removed the pre-loop invalidation; added a post-loop invalidation at the end of the method body.

### I6: `getMessagesForLLM` shared-reference JSDoc

**File**: `packages/sdk/src/_internal/platform/core/conversation.ts`

- Added `@returns readonly reference — do not mutate; the array is shared across cache-hit callers until the next conversation mutation` JSDoc comment above `getMessagesForLLM` to document the cache contract and prevent accidental caller mutation.

### I7: Dev-only undefined-slot assertion in ring accessor

**File**: `packages/sdk/src/_internal/platform/control-plane/gateway.ts`

- The `recentEvents` getter silently skipped `undefined` entries with `if (entry) out.push(entry)`. With correct `_recentEventsCount` tracking this path should never trigger.
- Added a `NODE_ENV !== 'production'` guarded `console.error` when an undefined slot is encountered, surfacing ring buffer accounting bugs during development. Production behavior unchanged (undefined slots are still skipped for resilience).

None. All changes are internal to private methods and fields; no public API surface changed.
