# 0.18.35 — Host binding port fix

**Released**: 2026-04-16
**Type**: Patch (bug fix)

## Summary

Fixes a regression in 0.18.34 where `resolveHostBinding` in `platform/daemon/host-resolver.ts` ignored the caller-configured port whenever `hostMode` was `'local'` or `'network'`. The 0.18.34 implementation forced every non-`custom` binding to the server-type default port (controlPlane: 3421, httpListener: 3422, web: 3423), which broke:

- Every daemon test that reserves a random high port (e.g., 39421) for test isolation
- Any deployment that configures a non-default port while keeping localhost-only binding
- Any environment that runs multiple daemon instances on different ports behind a reverse proxy

## Fix

Port is now always caller-controlled. `hostMode` decides only the bind address:

| Mode | Host | Port |
|------|------|------|
| `local` | 127.0.0.1 | configured port, or default if unset/zero |
| `network` | 0.0.0.0 | configured port, or default if unset/zero |
| `custom` | configured host, or 127.0.0.1 | configured port, or default if unset/zero |

This matches the original intent: `hostMode` separates localhost-only from network-exposed bindings, and both are configurable on any port.

## Files changed

- `packages/sdk/src/_internal/platform/daemon/host-resolver.ts`
- All workspace `package.json` files (version bump 0.18.34 → 0.18.35)
- `CHANGELOG.md`

## Migration

None. This reverts the port-hardcoding behavior introduced in 0.18.34 back to the behavior consumers expected (and that previous versions provided through direct port injection).
