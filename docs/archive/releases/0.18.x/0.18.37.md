# 0.18.37 — Companion pairing + bootstrap drift detection

**Released**: 2026-04-16
**Type**: Patch (bug fixes)

## Summary

Two pairing/auth issues surfaced by real companion-app scans:

1. The embedded daemon started by `startHostServices` was called with `daemon.enable({ daemon: true })` with no bearer token. Surfaces that issued companion-pairing QR tokens had no way to register those tokens with the daemon, so the phone's scanned bearer was silently rejected.
2. Manually editing `auth-bootstrap.txt` does not update `auth-users.json` (the bootstrap file is output-only), but there was no warning when the two drifted, so users would spend hours wondering why `/login` rejected the password they could clearly see in the file.

## What changed

### `packages/sdk/src/_internal/platform/runtime/bootstrap-services.ts`

- `ServiceFactories` gained two new optional fields: `sharedDaemonToken` and `sharedHttpListenerToken`
- `daemon.enable({ daemon: true }, factories.sharedDaemonToken)` — registers the token if provided
- `httpListener.enable({ httpListener: true }, factories.sharedHttpListenerToken)` — same for the HTTP listener

### `packages/sdk/src/_internal/platform/security/user-auth.ts`

- `UserAuthManager` constructor now calls `detectBootstrapCredentialDrift` after seeding users (only when `config.users` is unset so tests opt out)
- If `auth-bootstrap.txt` exists and its plaintext password does not verify against the hash in `auth-users.json`, `logger.warn` surfaces a clear diagnostic plus remediation steps
- Unknown username in bootstrap file produces a separate warning

## Migration

None. Both changes are additive.

Surfaces that advertise companion-pairing tokens should start passing `sharedDaemonToken` through the bootstrap factories:

```ts
const tokenRecord = getOrCreateCompanionToken('your-surface');
await startExternalServices(configManager, bus, dispatcher, {
  sharedDaemonToken: tokenRecord.token,
}, runtimeServices);
```
