# Release 0.18.42 — Daemon State Reconciliation

**Date**: 2026-04-16  
**Type**: Bug Fix  
**Affects**: `@pellux/goodvibes-sdk`, `@pellux/goodvibes-daemon-sdk`

## Problem

A live daemon at `192.168.0.61:3421` reported the following hung state after a session:

- 8 agent tasks stuck in `running` (never reached `completed`/`failed`/`cancelled`)
- 10 sessions all reported `active`; 9 of them were ghost sessions (`messageCount=0`)
- Input records stuck in `spawned` indefinitely
- `smsg` records never reaching terminal state

The root cause was two missing wires between the `RuntimeEventBus` and the state-tracking subsystems. `AgentManager.spawn()` runs agents asynchronously but emits no terminal bus events on completion — the bus events come from other codepaths. Both `AgentTaskAdapter.handleAgentStateChange()` and `SharedSessionBroker.finalizeAgentInputs()` had correct transition logic but were never subscribed to bus events, so state never progressed.

## Fixes

### δ1 — SharedSessionBroker: wire agent terminal events to input records

Added `attachRuntimeBus(bus: RuntimeEventBus, sessionResolver: (agentId: string) => string | null): () => void` to `SharedSessionBroker`.

Subscribes to `AGENT_COMPLETED`, `AGENT_FAILED`, and `AGENT_CANCELLED` on the `RuntimeEventBus`. On each event, resolves the session for the agentId (via the provided resolver) and calls `completeAgent()` with the mapped terminal status. Returns an unsubscribe function for clean teardown.

**Before**: `spawned` input records stuck forever after agent exit.  
**After**: Input records transition to `completed`/`failed`/`cancelled` within one event cycle of agent termination.

### δ2 — AgentTaskAdapter: wire agent terminal events to task registry

Added `attachRuntimeBus(bus: RuntimeEventBus): () => void` to `AgentTaskAdapter`.

Subscribes to the same three agent terminal events. On each event, looks up the taskId in `_agentToTask` map (no-op if the agentId is not tracked), then calls `handleAgentStateChange()` with the mapped state. Returns an unsubscribe function.

**Before**: Task registry rows permanently stuck in `running`.  
**After**: Tasks transition to terminal state when the corresponding agent emits a terminal event.

### δ3 — SharedSessionBroker: lastActivityAt field + idle-session GC sweep

Added `lastActivityAt: number` to `SharedSessionRecord` (updated in `createSession`, `bindAgent`, and `appendMessage`).

Added `_gcSweep()` private method called every 60 seconds (via `setInterval` started on first `start()` call). The sweep closes idle sessions that have no active agent:

| Policy | Condition | Reason |
|--------|-----------|--------|
| Empty-ghost | `messageCount === 0` and `idle >= idleEmptyMs` (default 10 min) | `idle-empty` |
| Long-idle | `messageCount > 0` and `idle >= idleLongMs` (default 24 h) | `idle-long` |

Sessions with `activeAgentId` set are immune to GC regardless of idle time. Closed sessions are persisted and a `session-closed` event is published with the reason.

**Before**: Ghost sessions (0-message) and abandoned sessions accumulated indefinitely.  
**After**: Ghost sessions expire after 10 minutes; long-idle sessions expire after 24 hours.

## Tests Added

`test/daemon-state-reconciliation.test.ts` — covers all three invariants:

- **DR2**: `AgentTaskAdapter.attachRuntimeBus` — COMPLETED/FAILED/CANCELLED transitions, unknown-agentId no-op guard, unsubscribe teardown
- **DR1**: `SharedSessionBroker.attachRuntimeBus` — input state transitions via bus events (spawned → completed/failed/cancelled), ghost-agentId guard
- **DR3**: `SharedSessionBroker` GC sweep — empty ghost close, content session safety at short idle, long-idle close with `idle-long` reason, active-agent immunity, event publishing
