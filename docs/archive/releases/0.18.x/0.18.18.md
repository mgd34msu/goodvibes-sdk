# 0.18.18

`0.18.18` completes the next SDK host-boundary cleanup pass and removes the last reusable package modules that were still silently constructing `.goodvibes/goodvibes/...` storage paths for themselves.

## Why This Release Exists

The previous host-boundary work removed the biggest hardcoded product-root assumptions, but a narrower set of reusable modules still had the same underlying problem:

- they were deciding that the host surface was always `goodvibes`
- they were constructing product-specific storage paths internally
- they forced host products to inherit those defaults instead of choosing their own surface layout

That is the wrong ownership boundary for the SDK. The SDK should define what needs to be configured. The host should decide how that configuration is supplied.

## What Changed

### Runtime State And Persistence

- `SessionManager`, session persistence helpers, WRFC workmaps, worktree registry state, and watcher persistence no longer bake in `.goodvibes/goodvibes/...` path segments.
- These modules now either:
  - accept host-owned scoped paths, or
  - fall back to the shared `.goodvibes/...` data root without inventing a product surface name.

### Tool Persistence Surfaces

- Team, worklist, packet, and query tool persistence no longer writes under `.goodvibes/goodvibes/...`.
- Tool registration now binds host-owned `surfaceRoot` where that ownership already exists.
- The SDK still exports compatibility defaults for direct use, but the reusable tool layer no longer hardcodes a product namespace.

### Agent And WRFC Runtime

- The agent orchestrator run context now carries `surfaceRoot`.
- Agent session state is now created from host-owned surface context instead of forcing the canonical product namespace.
- WRFC workmap persistence now resolves from host-owned surface state instead of constructing `goodvibes` paths locally.

### Daemon Service Manager

- `PlatformServiceManager` is now configurable for:
  - `surfaceRoot`
  - binary base name
  - default service name
  - default service description
- The reusable service manager no longer assumes the binary is always `goodvibes` or that service state always lives under `.goodvibes/goodvibes/service`.
- The canonical `goodvibes` defaults now live only in concrete daemon host composition, which is the correct ownership boundary.

### Catalog, Registry, And Intelligence Paths

- Ecosystem catalog receipts and manifests now use generic `.goodvibes/ecosystem/...` paths instead of `.goodvibes/goodvibes/ecosystem/...`.
- Registry skill/agent discovery and skill-loader fallback discovery no longer search product-specific `.goodvibes/goodvibes/...` paths.
- Intelligence language-config loading now resolves through host path services when available and otherwise falls back to shared `.goodvibes/languages`.

### Scheduler And Inline SDK Docs

- The cron helper state file no longer uses a product-specific path.
- Inline comments, help text, and schema descriptions were updated so the SDK no longer documents the old `.goodvibes/goodvibes/...` layout as if it were still current behavior.

## Boundary After This Release

After `0.18.18`, the reusable SDK code no longer contains live product-root construction for `goodvibes` except where that choice is intentionally made by the concrete daemon host composition layer.

That means:

- reusable SDK modules describe storage/config inputs
- host products choose surface names and storage layout
- canonical `goodvibes` remains a host decision, not a reusable SDK default

## Validation

- `bun x tsc -b packages/sdk --force --pretty false`
- `bun run validate`
- code-level search for reusable `.goodvibes/goodvibes/...` construction is clean except the deliberate concrete daemon host description in daemon facade composition
