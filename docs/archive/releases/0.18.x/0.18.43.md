# 0.18.43 — Companion chat + Problem 2 message routing + daemon-reconciliation fix-forward

**Released**: 2026-04-16
**Type**: Feature + correctness

## Summary

Coordinated release bundling two WRFC chains that each passed review at 10.0/10.0:

1. **Companion chat mode** (new): mobile/web companion apps can now open isolated SDK-side chat sessions via `/api/companion/chat/*` routes and stream orchestrator responses over SSE. Distinct from task-submission; no agent spawn.
2. **Companion message routing** (new, Problem 2): the existing session submit endpoint accepts `kind: 'task' | 'message'`. Message kind injects into the operator's TUI conversation as a system-tagged line via a scoped `conversation.followup.companion` event — no agent spawn, no orchestrator turn.
3. **Daemon state reconciliation**: the v0.18.42 fix was inert (the wiring was defined but never called). This release wires it up in default composition, adds startup reconciliation for pre-existing stuck records, plumbs a proper shutdown path, and widens `lastActivityAt` update coverage.

## User-visible changes

### Companion chat mode

New daemon routes (bearer-token auth, same as existing companion endpoints):

- `POST /api/companion/chat/sessions` — create a chat session. Returns `{ sessionId, createdAt }`.
- `POST /api/companion/chat/sessions/:id/messages` — submit a user message. Returns `{ messageId }`; the turn runs asynchronously and events stream via the SSE endpoint.
- `GET /api/companion/chat/sessions/:id/events` (SSE) — per-session event stream: `turn.started`, `turn.delta`, `turn.tool_call`, `turn.tool_result`, `turn.completed`, `turn.error`. Scoped by `clientId: "companion-chat:<sessionId>"` so no cross-session bleed.
- `GET /api/companion/chat/sessions/:id` — session metadata + full message history.
- `DELETE /api/companion/chat/sessions/:id` — close the session (aborts in-flight turn).

Chat sessions are isolated: their events never reach the TUI control-plane event feed, and chat-session data is not persisted across daemon restart in this version (in-memory only — planned for a follow-up patch).

### Companion message routing (Problem 2)

The existing `POST /api/sessions/:sessionId/submit` gains an optional `kind` field:

- `kind: 'task'` (default, unchanged) — spawns an agent via `SharedSessionBroker.bindAgent`.
- `kind: 'message'` — publishes a `conversation.followup.companion` event with a `ConversationMessageEnvelope` payload (`source: 'companion-followup'`) scoped to TUI subscribers only. No agent spawn, no input record in `spawned` state, no orchestrator turn. Response: `{ messageId, routedTo: 'conversation' }`.

TUI integration: subscribe to the event, append the body to the active `ConversationManager` as a system-tagged line `"[Companion] <body>"`. Operator decides whether to respond. Full client-side spec in `docs/companion-message-routing.md`.

### Shared abstraction

New `ConversationMessageEnvelope` type owned by `packages/sdk/src/_internal/platform/control-plane/conversation-message.ts`, used by both chat-mode events and Problem 2 events:

```ts
export type MessageSource =
  | 'operator'
  | 'companion-chat-user'
  | 'companion-chat-assistant'
  | 'companion-followup'
  | 'system'
  | 'tool';

export interface ConversationMessageEnvelope {
  readonly sessionId: string;
  readonly messageId: string;
  readonly body: string;
  readonly source: MessageSource;
  readonly timestamp: number;
  readonly metadata?: Readonly<Record<string, unknown>>;
}
```

Required (not optional) on `CompanionChatTurnStartedEvent`, `CompanionChatTurnCompletedEvent`, and `DaemonRuntimeRouteContext.publishConversationFollowup`. Missing composition paths produce compile errors, not silent drops.

### Daemon state reconciliation — four majors fixed

**M1 — production wiring**: `v0.18.42` defined `attachRuntimeBus` methods on `SharedSessionBroker` and `AgentTaskAdapter` but never called them in default composition. Users saw no change after upgrading. Now wired in `facade.ts` + `facade-composition.ts`; `AgentManager.spawn` invokes `wrapAgent(agentId, task, {sessionId})` so the adapter's map is populated.

**M2 — startup reconciliation**: `SessionBroker.start()` now reconciles pre-existing stuck `spawned`/`delivered` inputs to `cancelled` with a clear error message, and nulls any lingering `activeAgentId` on sessions. `AgentTaskAdapter.reconcileOnRestart()` marks running tasks as `aborted` with reason `daemon-restart`. Users upgrading will see stuck records unwedge on first boot.

**M3 — graceful shutdown**: GC interval `.unref()`'d so it doesn't pin the event loop on daemon shutdown. New `SessionBroker.stop()` clears the interval, invokes all bus unsubscribes, persists state. Wired into `DaemonServer.stop()`.

**M4 — `lastActivityAt` coverage**: new `_touch(sessionId)` helper called from all 7 mutation sites (`recordInput`, `updateInput`, `claimNextQueuedInput`, `finalizeAgentInputs`, `attachParticipantAndRoute`, `closeSession`, `completeAgent`). GC sweep now guards `if (session.pendingInputCount > 0) continue;` so sessions with pending follow-ups never close prematurely. `completeAgent` explicitly updates `lastActivityAt`.

**Hardening (m1-m4)**: bus subscribers `.catch`-wrapped; payload runtime guards on all bus handlers; `attachRuntimeBus` tracks `_busAttached` for idempotency with flag reset on unsub; `_gcSweep` single-pass `changed` tracking.

**Type-safety**: replaced `(input as any).kind` in `createSession` with a named scoped `CreateSessionInputWithKind` interface. JSDoc on `lastActivityAt` updated to match actual update coverage.

- 45 tests covering companion chat (routes, isolation, lifecycle) + companion message routing (envelope identity, `kind` validation, gateway filter scoping, oversized body)
- 29 tests covering daemon state reconciliation (M1 integration, M2 startup reconciliation both paths, M3 stop teardown, M4 `_touch` per mutation site + GC pending-input guard, m3 idempotency)
- Both chains reviewed at 10.0/10.0 via the WRFC loop. No known defects.

## Migration

Zero breaking changes. All new surface is additive. Consumers on 0.18.42 will pick up the reconciliation fixes automatically after upgrading — stuck records from prior daemon runs will be reconciled on first start.
