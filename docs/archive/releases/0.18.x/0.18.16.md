# 0.18.16

`0.18.16` fixes a host-boundary defect in the SDK itself.

The reusable SDK runtime/tool surfaces were still assuming the canonical `goodvibes` product root for persisted state. That was wrong for hosts like `goodvibes-tui`, which should decide their own surface-specific storage/config namespace while still consuming SDK-owned platform code.

## What Changed

### Host-owned surface roots in reusable SDK APIs

The reusable SDK APIs that TUI host code needs for runtime and tool composition no longer bake in a product-root default.

Changed surfaces:

- `createRuntimeServices(...)`
- `registerAllTools(...)`
- `createReplTool(...)`

Those surfaces now require host-owned `surfaceRoot` input instead of silently assuming `goodvibes`.

### Product defaults moved to concrete host wiring

The canonical `goodvibes` root is still used by the SDK’s own standalone daemon host, but that choice now lives in concrete host wiring instead of the reusable runtime/tool layer.

That keeps the boundary correct:

- SDK reusable platform code describes what needs to be configured
- host applications decide how their surface-specific storage is laid out

## Why This Matters

This patch is required for the TUI cutover.

Without it, switching the TUI to the SDK-owned REPL/runtime surfaces would send host-owned state into the wrong `.goodvibes/...` namespace and keep product-specific assumptions embedded in the reusable package.

After this patch:

- the SDK remains reusable across hosts
- the TUI can inject `tui` at the host boundary
- standalone daemon hosting can still inject `goodvibes`
