# 0.18.30

`0.18.30` adds the `platform/pairing/` module for QR-code-based mobile companion app pairing and introduces the `tools.llmEnabled` config key to make tool LLM usage explicitly opt-in.

## What Changed

### platform/pairing/ module

- Added `platform/pairing/` with QR code generation, companion token management, and connection info formatting for mobile companion app pairing
- QR generator shells out to `qrencode` CLI when available, falls back to a built-in encoder for QR versions 1-10
- Companion tokens persist to `.goodvibes/<surface>/companion-token.json` with create/regenerate/invalidate lifecycle
- Connection info formatter produces human-readable daemon startup blocks with embedded QR codes

### tools.llmEnabled config key

- Added `tools.llmEnabled` config key (default: `false`) — tool LLM is now explicitly opt-in, matching the existing `helper.enabled` behavior
- `resolveToolLLM()` now checks `tools.llmEnabled` before any provider resolution — when disabled, tool LLM calls return an empty string instead of silently falling through to the main conversation model

## Why This Patch Exists

The pairing module gives SDK hosts a reusable, surface-scoped companion pairing surface so they do not need to build QR generation and token lifecycle management from scratch. The `tools.llmEnabled` gate closes a silent footgun where tool LLM calls would fall through to the main conversation model when no dedicated tool provider was configured, making the opt-in boundary explicit and consistent with the rest of the optional helper configuration.
