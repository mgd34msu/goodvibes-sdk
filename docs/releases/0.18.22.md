# 0.18.22

`0.18.22` finishes the next SDK host-boundary cleanup pass by removing copied TUI runtime classes from the reusable package surface and by eliminating the last internal `tui` fallback from automation session routing.

## Why This Release Exists

Two boundary problems were still present in the published SDK:

- the runtime layer still carried copied concrete host UI classes for command registration, keybindings, and panel management
- the automation manager still defaulted missing host surface configuration to `tui` for `main` and `current` execution targets

Both are wrong for a reusable SDK.

The SDK can define the runtime contracts and shared behavior, but the host must own:

- which surface is active
- how command registries and keybindings are implemented
- how panel/workspace state is managed

## What Changed

### Host UI Boundary Cleanup

The SDK no longer ships copied concrete implementations of:

- command registry
- keybindings manager
- panel manager
- panel/pane workspace types

Those classes were TUI host code, not reusable SDK internals.

The runtime now depends on explicit host-facing interfaces in:

- `platform/runtime/host-ui`

That keeps the reusable package focused on runtime/domain logic while making the host supply its own concrete UI behavior.

### Runtime Services Composition

`createRuntimeServices(...)` no longer constructs TUI-flavored panel and keybinding managers internally.

Instead it now:

- accepts host-supplied panel/keybinding managers
- uses neutral no-op defaults when the host does not provide them

That makes runtime composition safe for non-TUI consumers and prevents the SDK from smuggling terminal UI assumptions into package consumers.

### Automation Surface Defaults

`main` and `current` automation targets no longer silently fall back to `tui`.

The automation execution layer now requires one of:

- an explicit `target.surfaceKind`
- or an explicit host `defaultSurfaceKind` provided by the caller

The canonical runtime services composition passes its host surface explicitly, which preserves the normal GoodVibes behavior without baking that choice into the reusable automation package.

## Downstream Impact

This patch is required for the continued TUI cutover because it removes one of the last SDK-owned places where host policy was still being invented inside the package.

That means:

- the SDK no longer owns TUI-specific UI implementations
- the SDK no longer decides that an unspecified host surface means `tui`
- the TUI can stay the host while the SDK stays reusable

- `bun run validate`
- `bun scripts/pack-check.ts`
