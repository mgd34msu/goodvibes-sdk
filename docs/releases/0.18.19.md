# 0.18.19

`0.18.19` fixes a package-surface typing mistake in the sandbox provisioning APIs that showed up immediately when the TUI switched from local sandbox provisioning code to the canonical SDK package exports.

## Why This Release Exists

The previous release published the sandbox runtime modules that the TUI needed to import directly. That package surface was real, but one part of it was still typed incorrectly:

- the sandbox provisioning module exported functions that accepted the SDK's concrete `ConfigManager` class
- a consuming host like `goodvibes-tui` has its own config-manager implementation and should not need to satisfy the SDK class identity
- because the SDK type used a private class member, the package consumer could not pass its compatible host config manager through normal typing

That is the wrong boundary. The provisioning layer should depend on configuration behavior, not on a concrete SDK-owned class instance.

## What Changed

### Sandbox Provisioning Public Types

- `buildSandboxDoctorChecks`
- `renderSandboxDoctor`
- `exportSandboxGuestBundle`
- `scaffoldSandboxQemuInitBundle`
- `scaffoldSandboxQemuSetupBundle`

These APIs now accept `ConfigManagerLike` instead of the SDK's concrete `ConfigManager` class.

### Writable Provisioning Flows

- `bootstrapSandboxQemuSetupBundle`
- `applySandboxQemuSetupManifest`

These APIs now accept `WritableConfigManagerLike`, which requires the behavior they actually need:

- `get(key)`
- `setDynamic(key, value)`

That keeps the package reusable while still expressing the write contract explicitly.

## Why This Boundary Is Correct

The SDK is supposed to define:

- what values the host needs to provide
- what behavior a provisioning-capable config manager must support

The SDK is not supposed to require:

- that the host instantiate the SDK's own config-manager class
- that a consuming product mirror SDK internals just to satisfy nominal typing

After this release, the sandbox provisioning package surface is behavior-based again instead of SDK-class-based.

## TUI Impact

This patch unblocks the TUI from:

- importing the canonical sandbox provisioning package surface
- deleting its local sandbox provisioning implementation
- compiling against the published SDK without type-casting around a private SDK class boundary
