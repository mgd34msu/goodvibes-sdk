# 0.18.44 — hostMode auto-restart behavior

**Released**: 2026-04-16
**Type**: Feature + reliability

## Summary

Daemon and HTTP listener now watch `controlPlane.hostMode/host/port` and
`httpListener.hostMode/host/port` config changes and transparently restart the affected
server so runtime config edits take effect without a full daemon bounce.

Underpinning this is a new keyed subscription API on `ConfigManager` and an idempotent
watcher helper that both servers share.

## Changes

### `ConfigManager.subscribe(key, cb)` — keyed listener API

New synchronous subscribe method. Accepts a config key and a callback invoked on every
change to that key. Each callback is wrapped in a `try/catch` so a failing listener never
blocks others.

Newly exported types:

- `ConfigChangeCallback<K>` — `(key: K, next: ConfigValue<K>, prev: ConfigValue<K>) => void`
- `ConfigUnsubscribe` — zero-argument disposer returned by `subscribe`

### `daemon/host-mode-watcher.ts` — `createHostModeRestartWatcher`

New internal helper. Accepts a list of config keys, a `getIsRunning` gate, and a restart
callback. Fans out subscriptions across all keys, debounces concurrent triggers into a
single restart invocation, and exposes an idempotent `unsubscribe()` that tears down all
listeners in one call. Both `DaemonServer` and `HttpListener` consume it.

### `DaemonServer` + `HttpListener` — transparent restart on config change

Both classes subscribe to `controlPlane.{hostMode,host,port}` and
`httpListener.{hostMode,host,port}` respectively via the watcher. When a watched key
changes while the server is running, the class:

1. Checks `_restarting` re-entrancy guard — if a restart is already in flight, sets
   `_restartDirty` and returns immediately (post-start loop picks it up).
2. Calls `stop()` then `start()` internally.
3. After `start()` completes, checks `_restartDirty` — if set, loops back into another
   restart cycle (`waitForRestart()` while-loop chaining).

The watcher instance is preserved across `stop()`/`start()` cycles so mid-restart config
changes are never dropped.

### `requirePortAvailable` — production-only gate

Both classes gate `requirePortAvailable` on `serveFactory === Bun.serve` so injected
mock server factories skip the real OS bind check during tests.

## Testing

19 tests covering:

- **CM1–CM4**: `ConfigManager` subscription notification, `try/catch` isolation per
  callback, multiple-key fan-out, and unsubscribe correctness.
- **HL1–HL6**: `HttpListener` listener lifecycle, re-entrancy guard, dirty-flag loop,
  mid-restart config change, unsubscribe-on-stop.
- **HW1–HW5**: `createHostModeRestartWatcher` helper — gate behavior, multi-key
  fan-out, idempotent unsubscribe.

WRFC chain passed at 10.0/10.0 after 3 review rounds.

## Breaking behavior

None. `ConfigManager.subscribe` is an additive method. Existing consumers that never
call it are unaffected.

## Migration

No migration steps. Servers that previously required a full daemon bounce after changing
`hostMode`, `host`, or `port` will now restart automatically.
