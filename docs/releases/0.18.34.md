# 0.18.34

`0.18.34` fixes five critical correctness bugs: an asymmetric daemon lifecycle, a mutable-Set iteration hazard in the event bus, a non-atomic session file write, an unbounded rate-limiter Map, and missing fetch timeouts on all non-streaming outbound HTTP calls.

## What Changed

- **C1 — Daemon shutdown (facade.ts)**: `DaemonServer.stop()` now tears down all services started during `start()`. Previously the method was effectively synchronous fire-and-forget, leaving the heartbeat watcher, reply poller, approval broker subscription, and HTTP router running until GC. The updated implementation stops them in reverse start order before closing the server socket.
- **C3 — Event bus snapshot (events/index.ts)**: `RuntimeEventBus.emit` now snapshots both the per-type listener Set and the per-domain listener Set into arrays before iterating. Previously, a listener that called `subscribe` or `unsubscribe` during dispatch could silently skip subsequent handlers in the same emit cycle.
- **C4 — Atomic session writes (sessions/manager.ts)**: `SessionManager` session writes now use a tmp-file + `fsync` + `rename` pattern via a new `_atomicWrite` helper. A process crash mid-write previously left an empty or partially written session file. An `_cleanupOrphanTempFiles` sweep runs at constructor time to remove `.tmp-*` files left by prior crashes.
- **C5 — Rate limiter TTL + LRU cap (http-listener.ts)**: `RateLimiter` now evicts entries older than 10 minutes (TTL), caps the Map at 10,000 entries with LRU eviction on overflow, runs a periodic 60-second sweep, and exposes a `stop()` method to cancel the sweep interval. `HttpListener.stop()` calls `this.rateLimiter.stop()` as its first action.
- **C6 — Fetch timeouts (fetch-with-timeout.ts + integrations)**: A new `fetchWithTimeout` utility wraps `fetch` with a 30-second default timeout using `AbortController` and `AbortSignal.any()` to merge the caller's own signal when present. Applied to all non-streaming outbound calls in `github.ts`, `ntfy.ts` (`publish` only — `poll` and `subscribeJsonStream` retain their caller-managed signals), `slack.ts` (all four call sites), the Slack response-URL handler in `adapters/slack/index.ts` (3 call sites), and `providers/gemini.ts` (cache create/delete — streaming calls are excluded).

## Why This Patch Exists

All five bugs were silent correctness hazards: no crashes, no test failures, but each one could corrupt state or consume unbounded resources in production under sustained load or adversarial conditions. C4 (session corruption) and C5 (memory growth) were the highest-urgency items in long-running daemon deployments. C1 and C3 could produce subtle behaviour differences under concurrent load. C6 eliminates the possibility of a hung daemon caused by an unresponsive upstream service holding an open connection indefinitely.
