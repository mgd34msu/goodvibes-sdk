# 0.18.14

First full public release of `@pellux/goodvibes-sdk`.

This release is the point where the SDK stops being a packaging experiment and becomes the canonical published GoodVibes TypeScript SDK:

- one public npm package
- stable subpath exports
- standalone install and validation
- reusable daemon/server integration layer
- extracted platform/runtime modules
- browser, Node, Bun, web UI, React Native, and Expo entrypoints
- release automation and package verification

## Package Model

This release publishes one public package:

- `@pellux/goodvibes-sdk`

That package exposes stable subpath entrypoints instead of requiring consumers to install separate sibling packages. The public install model is now:

- one package in `package.json`
- import only the entrypoints you need
- no nested public internal-package tree

- `@pellux/goodvibes-sdk`

## Daemon And Server Integration Layer

This release includes the reusable daemon/server route layer, not just client code.

Included daemon/server surfaces:

- control routes
- telemetry routes
- runtime routes
- runtime session routes
- runtime automation routes
- remote routes
- channel routes
- integration routes
- knowledge routes
- media routes
- system routes
- operator/task/session/automation/remote dispatchers
- daemon auth/policy helpers
- structured daemon error-response helpers
- route helper/context contracts

The intent is that TypeScript hosts can embed the GoodVibes daemon route layer without having to reimplement route registration semantics, error normalization, or handler wiring themselves.

## Packaging And Install Shape

This release finalizes the install shape:

- one public package
- flattened published artifact
- public subpath exports
- no leaked nested internal workspace packages in the installed package
- no runtime dependency on sibling workspace packages
- install-smoke validation that checks the published package shape directly

This release also locks in a public export for:

- `@pellux/goodvibes-sdk/platform/runtime/inspection/state-inspector`

That export exists so hosts can depend on the SDK-owned state-inspector implementation directly instead of carrying local copies.
